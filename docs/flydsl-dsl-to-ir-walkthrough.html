1. 整体流程概览:从 Python 函数到 MLIR Module

以 vector_add_kernel 为例,用户编写一个 Python 函数并加上 @flyc.kernel 装饰器.当它被 @flyc.jit 装饰的函数调用时,FlyDSL 编译器执行以下步骤:

2. AST 改写:5 个 Transformer 的前后对比

ASTRewriter.transform(func) 在装饰时执行----@flyc.kernel 和 @flyc.jit 分别在各自的 __init__ 中调用它.它通过 inspect.getsource() 获取函数源码,ast.parse() 解析为 AST,然后依次应用 5 个注册的 Transformer,最终 compile() 回 Python 字节码并替换 func.__code__.

2.1 Transformer 1:RewriteBoolOps -- 布尔运算改写

目的:将 Python 的 and/or/not 改写为 DSL 版本 dsl_and_/dsl_or_/dsl_not_,使其支持 MLIR Value 的短路求值.

改写前

if a and b:
    ...
result = not flag

改写后

if dsl_and_(a, b) if not (isinstance(a, bool) and a == False) else a:
    ...
result = dsl_not_(flag)

dsl_and_ 会先检查对象是否有 __fly_and__ 方法.有这个方法的是 Numeric 子类(如 Int32,Boolean),因为它们可能代表一个 MLIR 动态值(编译期未知),需要生成 arith.andi Op;而 Tensor,Layout 等 DSL 类型没有 __fly_and__,对它们做布尔与没有意义,会回退到 Python 原生 and.这保证了编译期和运行时的兼容性.

2.2 Transformer 2:ReplaceIfWithDispatch -- if/else 改写为 scf.if

目的:将 Python if/else 改写为 scf_if_dispatch 调用,在运行时根据条件是否为 MLIR Value 来决定走 scf.if 还是 Python 原生 if.

改写前

if cond:
    x = a + b
    y = c + d
else:
    x = a - b
    y = c - d

改写后

def __then_0(__ret_names, x, y):
    x = a + b
    y = c + d
    return scf_if_collect_results(__ret_names, locals())

def __else_0(__ret_names, x, y):
    x = a - b
    y = c - d
    return scf_if_collect_results(__ret_names, locals())

x, y = scf_if_dispatch(cond, __then_0, __else_0,
                        result_names=('x', 'y'),
                        result_values=(locals().get('x', None), locals().get('y', None)))

关键机制:

  • _collect_assigned_vars 分析 if/else body 中赋值的变量,确定需要从 scf.if 返回的 result_names
  • then/else 分支被提取为独立函数,函数体末尾追加 scf_if_collect_results 收集结果
  • scf_if_dispatch 在运行时判断:若 cond 是 ir.Value(动态条件),生成 scf.IfOp;否则走 Python 原生 if
  • 若 cond 被 const_expr() 包裹,则跳过改写,保持 Python 原生 if

2.3 Transformer 3:InsertEmptyYieldForSCFFor -- for 循环改写为 scf.for

目的:将 Python for i in range(...) 改写为 scf_for_dispatch 调用,在运行时生成 scf.ForOp.

改写前

for i in range(n):
    acc = acc + data[i]

改写后

def __for_body_0(i, __ret_names, acc):
    acc = acc + data[i]
    return scf_for_collect_results(__ret_names, locals())

acc = scf_for_dispatch(0, n, 1, __for_body_0,
                        result_names=('acc',),
                        result_values=(locals().get('acc', None),))

关键机制:

  • 自动推断循环携带变量(loop-carried values):分析 body 中赋值且在循环前已存在的变量
  • scf_for_dispatch 在运行时生成 scf.ForOp,循环携带变量作为 iter_args
  • 若 range 参数是 const_expr(),则跳过改写,保持 Python 原生 for
  • 也支持 scf_range 形式的手动 yield 循环

2.4 Transformer 4:ReplaceYieldWithSCFYield -- yield 改写为 scf.yield

目的:将 Python yield 语句改写为 scf_yield_() 调用,在运行时生成 scf.YieldOp.

改写前

for i in scf_range(0, n, 1, init=[acc]):
    yield acc

改写后

for i in scf_range(0, n, 1, init=[acc]):
    scf_yield_(acc)

scf_yield_ 在运行时调用 scf.YieldOp 生成 MLIR yield 操作,并返回父 Op 的 results(如果有的话).

2.5 Transformer 5:CanonicalizeWhile -- while 循环改写为 scf.while

目的:将 Python while 循环改写为 scf_while_gen 调用,在运行时生成 scf.WhileOp.

改写前

while cond:
    body

改写后

w_42 = scf_while_gen(cond)
while (__init__42 := next(w_42, False)):
    body

scf_while_gen 是一个生成器,第一次 next() 创建 scf.WhileOp 并设置 before/after region,第二次 next() 返回 False 退出.循环体在 after region 的 InsertionPoint 中执行.

2.6 Transformer 应用顺序与全局注入

5 个 Transformer 按 @ASTRewriter.register 的注册顺序依次应用:

顺序Transformer注入的全局函数
1RewriteBoolOpsdsl_and_, dsl_or_, dsl_not_
2ReplaceIfWithDispatchconst_expr, scf_if_dispatch, scf_ifexp_dispatch, scf_if_collect_results
3InsertEmptyYieldForSCFForscf_range, scf_for_dispatch, scf_for_collect_results
4ReplaceYieldWithSCFYieldscf_yield_
5CanonicalizeWhilescf_while_gen, scf_while_init

每个 Transformer 的 rewrite_globals() 方法返回需要注入到函数全局作用域的函数.改写完成后,ASTRewriter.transform 通过 f.__globals__[name] = val 将这些函数注入,使得改写后的代码可以引用它们.

2.7 vectorAdd 为什么不需要 AST 改写?

vector_add_kernel 中没有 if/else/for/while 等控制流,也没有布尔运算,所以 5 个 Transformer 都不会修改它的 AST.改写后的代码与原始代码完全相同.但 AST 改写机制仍然会执行(获取源码,解析,遍历,编译),只是不会产生实质变化.

2.8 改写前后对比:含控制流的 kernel 示例

当一个 kernel 包含 if/for/and 等控制流时,5 个 Transformer 按 Sec.2.6 的顺序依次改写 AST.下面是一个嵌套示例(改写后代码已省略部分细节,完整形态见 Sec.2.2 与 Sec.2.3):

改写前(用户写的代码) 改写后(self._func 指向的版本,示意) 
@flyc.kernel
def example(cond, acc):
    if cond and acc > 0:
        for i in range(10):
            acc = acc + 1
 
@flyc.kernel
def example(cond, acc):
    def __then_0(__ret_names, acc):
        def __for_body_0(i, __ret_names, acc):
            acc = acc + 1
            return scf_for_collect_results(__ret_names, locals())
        acc = scf_for_dispatch(0, 10, 1, __for_body_0,
            result_names=('acc',),
            result_values=(locals().get('acc', None),))
        return scf_if_collect_results(__ret_names, locals())

    acc = scf_if_dispatch(
        dsl_and_(cond, acc > 0) if not (isinstance(cond, bool) and cond == False) else cond,
        __then_0, __else_0,
        result_names=('acc',),
        result_values=(locals().get('acc', None),))

三个 Transformer 各自负责一类改写(名称与 Sec.2.6 一致):

  • RewriteBoolOps:cond and acc > 0 --> dsl_and_(cond, acc > 0)(运行时若操作数为 Numeric,经 __fly_and__ 生成 arith.andi;否则回退 Python and)
  • ReplaceIfWithDispatch:if ...: --> scf_if_dispatch(cond, __then_0, __else_0, result_names=..., result_values=...)
  • InsertEmptyYieldForSCFFor:for i in range(10): --> scf_for_dispatch(0, 10, 1, __for_body_0, result_names=..., result_values=...)

注意:Python 的按位与 a & b(ast.BitAnd)不会被 AST 改写;只有逻辑与 a and b 才会变成 dsl_and_.改写后的代码在 tracing 执行时,scf_if_dispatch/scf_for_dispatch 在条件/循环边界为动态 MLIR Value 时发射 scf.if/scf.for,否则走 Python 原生分支.

3. Kernel 发射:从改写后的函数到 gpu.func

AST 改写发生在装饰时,而 gpu.func 的创建发生在 kernel 被调用时(即 @flyc.jit 函数体内调用 vector_add_kernel(a, c, block_dim) 时).

3.1 JIT 函数的调用入口

当用户调用 vector_add(ta, c, n, n+1) 时:

# jit_function.py 核心流程(简化)
class JITFunction:
    def __call__(self, *args, **kwargs):
        # 1. 创建 MLIR Context 和 Module
        with _create_mlir_context() as ctx:
            module = ir.Module.create()
            # 2. 创建 gpu.module 与 func.func(host)
            with ir.InsertionPoint(module.body):
                gpu_module = create_gpu_module("kernels")
            func_op = func.FuncOp(self.func.__name__, (ir_types, []))
            entry_block = func_op.add_entry_block()

            # 3. 在 CompilationContext 中执行 host 函数(tracing)
            with CompilationContext.create() as comp_ctx:
                comp_ctx.gpu_module_body = get_gpu_module_body(gpu_module)
                with ir.InsertionPoint(entry_block):
                    dsl_args = construct_from_ir_values(dsl_types, jit_args, ir_args)
                    self.func(**named_args)   # 内部调用 kernel --> 触发 3.2
                    func.ReturnOp([])

            # 4. 保存 origin IR,再运行 Pass Pipeline
            original_ir = module.operation.get_asm(enable_debug_info=True)
            compiled_module = MlirCompiler.compile(
                module, arch=backend.target.arch, func_name=self.func.__name__
            )

            # 5. 构造 CompiledArtifact 并执行(或写入 cache)
            compiled_func = CompiledArtifact(compiled_module, self.func.__name__, original_ir)
            return compiled_func(args_tuple)
3.1 与 3.2 的关系

3.1 的 self.func(**named_args) 执行 vector_add 函数体,函数体内调用 vector_add_kernel(a, c, block_dim) 时触发 KernelFunction.__call__,即进入 3.2,嵌套在 tracing 阶段(步骤 4 之前)执行.Pass Pipeline 在 tracing 全部完成、func.ReturnOp 之后,由 MlirCompiler.compile 统一运行.

3.2 Kernel 函数的发射过程

3.1 中 self.func() 执行到 vector_add_kernel(a, c, block_dim) 时,KernelFunction.__call__ 被触发:

# kernel_function.py KernelFunction._emit_kernel(简化)
def _emit_kernel(self, ctx, args, kwargs):
    # 1. 解析参数,区分 constexpr 和普通参数
    #    a: Tensor, c: Tensor --> 普通参数
    #    block_dim: Constexpr[int] --> constexpr 参数

    # 2. 获取参数的 MLIR 类型
    kernel_arg_types = []
    for value in param_values:
        kernel_arg_types.extend(get_ir_types(value))
    # --> [TensorType(f32, ...), TensorType(f32, ...)]

    # 3. 创建 gpu.func
    with ir.InsertionPoint(ctx.gpu_module_body):
        func_type = ir.FunctionType.get(kernel_arg_types, [])
        gpu_func = create_gpu_func(kernel_name, ir.TypeAttr.get(func_type))
        gpu_func.regions[0].blocks.append(*kernel_arg_types)
        entry_block = gpu_func.regions[0].blocks[0]

        # 4. 在 entry_block 中设置 InsertionPoint
        with ir.InsertionPoint(entry_block), kernel_loc:
            # 5. 将 Block 参数重建为 DSL 对象
            block_args = list(entry_block.arguments)
            dsl_args = {}
            idx = 0
            for param_name, value in zip(param_names, param_values):
                n = len(get_ir_types(value))
                dsl_args[param_name] = construct_from_ir_values(
                    type(value), value, list(block_args[idx:idx+n])
                )
                idx += n
            dsl_args.update(constexpr_values)

            # 6. 执行步骤 1 已改写好的函数体 --> 触发 tracing
            self._func(**dsl_args)
            gpu.ReturnOp([])

3.3 参数重建:construct_from_ir_values

gpu.func 的 Block 参数是 ir.Value,但步骤 1 已改写好的函数体期望接收 DSL 对象(如 Tensor).protocol.construct_from_ir_values 负责分派到各类型的 __construct_from_ir_values__:

# protocol.py
def construct_from_ir_values(dsl_type, args, values):
    if hasattr(dsl_type, "__construct_from_ir_values__"):
        return dsl_type.__construct_from_ir_values__(values)
    ...

# expr/typing.py BuiltinDslType 的实现
class BuiltinDslType(ir.Value):
    @classmethod
    def __construct_from_ir_values__(cls, values):
        return cls(values[0])

# 例如:Tensor.__construct_from_ir_values__([ir.Value(%arg0)])
# --> Tensor(ir.Value(%arg0))

这样,self._func(a=Tensor(...), c=Tensor(...), block_dim=64) 被调用时,函数体内的 fx.logical_divide(a, ...) 等调用就能正常工作.

3.4 Constexpr 参数:编译期常量不进入 gpu.func 签名

在 vector_add_kernel 中,block_dim: fx.Constexpr[int] 是一个 Constexpr 参数.它的处理路径与普通参数(a, c)完全不同:

# _emit_kernel 中的参数分流逻辑
for param_name, value in bound.arguments.items():
    param = sig.parameters[param_name]
    annotation = param.annotation
    if Constexpr.is_constexpr_annotation(annotation):
        constexpr_values[param_name] = value    # <-- block_dim 走这里
    else:
        param_names.append(param_name)           # <-- a, c 走这里
        param_values.append(value)

分流后,两类参数的命运截然不同:

参数类型是否进入 gpu.func 签名传入函数体的方式
aTensor-> %arg0: !fly.Tensor<...>Block 参数 -> construct_from_ir_values -> DSL 对象
cTensor-> %arg1: !fly.Tensor<...>Block 参数 -> construct_from_ir_values -> DSL 对象
block_dimConstexpr[int]不进入直接作为 Python int 64 传入

具体来说,constexpr_values 通过 dsl_args.update(constexpr_values) 直接合并到参数字典中,不经过 get_ir_types --> Block 参数 --> construct_from_ir_values 这条路.

为什么 Constexpr 不进入 gpu.func 签名?

Constexpr 的语义是"编译期常量"----它的值在 tracing 时就确定了,不需要在 GPU kernel 运行时从参数传入.这样做有两个关键好处:

  • 常量折叠:编译器可以看到 block_dim=64 的具体值,在 IR 中直接生成 arith.constant 64 : i32,而非运行时参数,从而允许后续 Pass 进行常量折叠和死代码消除.
  • Layout 静态推导:fx.make_layout(block_dim, 1) 中 block_dim 是已知常量,编译器可以静态推导出完整的 Layout 形状,而无需在运行时动态计算.

如果 block_dim 是运行时参数,这些优化都无法进行.

在函数体执行时,fx.make_layout(block_dim, 1) 中的 block_dim 就是 Python 原生 int 64.它会在 make_layout 内部被物化为 MLIR 常量(如 arith.constant 64 : i32),而非从 kernel 参数传入.

3.5 Kernel 启动:从 KernelLauncher 到 gpu.launch_func

前面的 3.2~3.4 讲的是 kernel 定义----在 gpu.module 内创建 gpu.func 并通过 tracing 填入 IR.但 gpu.func 本身不会执行,它需要被 启动 才能在 GPU 上运行.

这就是 KernelLauncher 的职责.用户代码中的调用链是:

# 01-vectorAdd.py
vector_add_kernel(a, c, block_dim)       # <-- 阶段一:发射 gpu.func(定义 kernel)
    .launch(grid=(grid_x,1,1),           # <-- 阶段二:发射 gpu.launch_func(启动 kernel)
            block=[block_dim,1,1],
            stream=stream)

KernelFunction.__call__ 返回一个 KernelLauncher 对象,它持有 kernel 名称和参数.当用户调用 .launch() 时,KernelLauncher.launch() 发射 gpu.launch_func:

# kernel_function.py KernelLauncher.launch(简化)
def launch(self, *, grid, block, stream=None, ...):
    # 1. 解包 kernel 参数为 ir.Value
    kernel_operands = []
    for arg in self._kernel_args:
        kernel_operands.extend(extract_to_ir_values(arg))

    # 2. 将 grid/block 维度转为 index 值
    grid_x, grid_y, grid_z = _to_index_value(...)
    block_x, block_y, block_z = _to_index_value(...)

    # 3. 发射 gpu.launch_func
    launch_op = gpu.LaunchFuncOp(
        ["kernels", self._kernel_name],   # 引用已定义的 gpu.func
        (grid_x, grid_y, grid_z),         # grid 维度
        (block_x, block_y, block_z),      # block 维度
        kernel_operands,                   # 传入参数
        ...
    )
Kernel 定义 vs Kernel 启动:两个不同的 IR 区域

gpu.func 和 gpu.launch_func 生成在 IR 的不同位置:

Kernel 定义Kernel 启动
IR 操作gpu.func @vector_add_kernel_0gpu.launch_func @kernels::vector_add_kernel_0
所在区域gpu.module 内部func.func(host 函数)内部
触发方式vector_add_kernel(a, c, block_dim).launch(grid=..., block=...)
语义"这个 kernel 做什么"(函数体 IR)"以什么配置运行这个 kernel"(启动参数)

这种分离与 CUDA 编程模型一致:__global__ void kernel(...) 定义 kernel,kernel<<<grid, block>>>(...) 启动 kernel.

4. traced_op:DSL 调用如何实时插入 IR

前两节我们看到了 AST 改写和 Kernel 发射的过程.本节深入回答一个核心问题:DSL 函数调用时,IR 是在哪里,如何被实时插入到 MLIR Module 中的?

4.1 核心结论:traced_op 搭台,OpView 唱戏

traced_op 本身不插入 IR,它只是一个"搭台"的包装器----设置调试位置,解包参数.真正插入 IR 的是它所包装的函数体内调用的 MLIR Op 构造器(OpView.__init__).

以 fx.logical_divide(a, layout) 为例,完整调用链如下:

用户代码: fx.logical_divide(a, layout) | v 1. traced_op 包装层(搭台) | _caller_location() --> 创建 ir.Location | _flatten_args() --> 解包 DSL 对象为 ir.Value | with loc: --> 设置 Location 上下文 | v 2. DSL 函数体(primitive.py) | return fly.logical_divide(layout, divisor, loc=loc, ip=ip) | v 3. MLIR ODS 自动生成的 Python 绑定 | return LogicalDivideOp(..., loc=loc, ip=ip).result | v 4. OpView.__init__()(C++ nanobind 绑定) | 读取 ir.InsertionPoint.current --> 确定插入位置 | 读取 ir.Location.current --> 确定调试位置 | 创建 Op --> 插入到 Block 中 | v 5. 返回 ir.Value(OpResult)

4.2 第一层:traced_op -- 搭台(设置上下文 + 解包参数)

traced_op 定义在 meta.py 中:

def traced_op(op):
    @wraps(op)
    def wrapper(*args, **kwargs):
        loc = kwargs.pop("loc", None)
        if loc is None:
            loc = _caller_location(depth=1)     # 1. 自动推断源码位置
        args, kwargs = _flatten_args(args, kwargs)  # 2. 解包 DSL 对象
        with loc:                                # 3. 设置 Location 上下文
            return op(*args, **kwargs)           # 4. 调用真正的函数体
    return wrapper

它做了三件关键的事:

4.2.1 _caller_location(depth=1) -- 自动推断源码位置

def _caller_location(depth=1):
    frame = inspect.currentframe()
    for _ in range(depth + 1):
        if frame is not None:
            frame = frame.f_back
    info = inspect.getframeinfo(frame)
    line = info.lineno
    col = info.col_offset or 0
    file_loc = ir.Location.file(info.filename, line, col)

    if info.code_context:
        label = " ".join(ln.strip() for ln in info.code_context)
    else:
        label = info.function
    return ir.Location.name(label, childLoc=file_loc)

这会生成一个复合 Location,例如:

loc("ta = fx.logical_divide(a, fx.make_layout(block_dim, 1))" at "01-vectorAdd.py":28:4)

外层是 NameLoc(包含源码文本),内层是 FileLineColLoc(文件名+行号+列号).这样在调试 IR 时可以直接看到每条 Op 对应的源代码.

4.2.2 _flatten_args() -- 解包 DSL 对象为 ir.Value

为什么需要解包? 因为 MLIR Op 构造器(如 LogicalDivideOp.__init__)只接受 ir.Value,不接受 Tensor,Int32 等 DSL 包装对象.__extract_to_ir_values__ 协议就是桥梁.

def _to_raw_value(obj):
    if isinstance(obj, ir.Value):
        return obj                          # 已经是 ir.Value,直接用
    if isinstance(obj, type):
        return obj                          # 类型对象(如 fx.Float32),直接用
    if hasattr(obj, "__extract_to_ir_values__"):
        values = obj.__extract_to_ir_values__()  # 调用协议方法解包
        if len(values) != 1:
            raise ValueError(...)
        return values[0]
    if isinstance(obj, (tuple, list)):
        return tuple(_to_raw_value(e) for e in obj)
    return obj                              # 其他(int, None 等),原样返回

两类 DSL 对象的解包方式不同:

DSL 类型基类__extract_to_ir_values__ 实现原因
Tensor, Layout, IntTuple, CopyAtom 等BuiltinDslType(ir.Value)return [self]自身就是 ir.Value 子类
Int32, Float32 等 NumericNumeric(普通 Python 类)return [self.ir_value()]需物化为 ir.Value
隐含的 IR 插入点
当 Int32(0) 或 Float32(3.14) 这样的 Numeric 对象被解包时,如果其内部值还是 Python 原生类型(int/float),ir_value() 会在此刻发射一个 arith.constant Op将其物化为 ir.Value.这也是 IR 插入的一个隐含点.

4.2.3 with loc: -- 设置 Location 上下文

ir.Location 实现了 __enter__/__exit__,进入 with 块后,ir.Location.current 返回这个 Location.后续所有 Op 构造器都会读取这个 Location 作为默认位置.

4.3 第二层:DSL 函数体 -- 调用 Fly 方言 Op

以 logical_divide 为例(primitive.py):

@traced_op
def logical_divide(layout, divisor, loc=None, ip=None):
    if not isinstance(divisor, ir.Value):
        divisor = make_tile(*divisor, loc=loc, ip=ip)
    return fly.logical_divide(layout, divisor, loc=loc, ip=ip)

fly 是 from .._mlir.dialects import fly,即 MLIR ODS 自动生成的 Python 绑定模块.

4.4 第三层:ODS 生成的 Python 绑定 -- 构造 Op 对象

fly.logical_divide 是 ODS 自动生成的代码:

def logical_divide(layout, divisor, *, results=None, loc=None, ip=None):
    return LogicalDivideOp(layout=layout, divisor=divisor,
                           results=results, loc=loc, ip=ip).result

LogicalDivideOp 也是自动生成的:

4.5 OpView 背景:MLIR Python 绑定中的 Operation 视图

在深入 OpView.__init__ 之前,有必要先理解 OpView 在 MLIR Python 绑定中的角色.

一句话概括:OpView 是每个 MLIR Operation 在 Python 侧的"视图对象"----它持有对 C++ mlir::Operation 的引用,并提供了 Python 友好的属性访问接口.

OpView 存在两层实现:

层级位置说明
C++ 绑定层 LLVM 上游 mlir/lib/Bindings/Python/IRCore.cpp 中的 PyOpView 类 用 nanobind 暴露给 Python,提供 operation 属性,__init__ 等
Python 生成层 构建时由 mlir-tblgen --gen-python-op-bindings 自动生成 每个具体 Op 继承 _ods_ir.OpView,生成 __init__,属性访问器等
注意:Python 生成层代码不在 Git 仓库中,只存在于 build 目录.例如:
  • build-fly/python_packages/flydsl/_mlir/dialects/_arith_ops_gen.py
  • build-fly/python/mlir_flydsl/dialects/_fly_ops_gen.py
它们由 .td(TableGen)定义驱动,在 CMake 构建时自动生成.

OpView 的核心职责可以用下图表示:

  2. 属性访问器(@property):如 .lhs,.result,从底层 mlir::Operation 的 operands/results 中读取对应的 Value
  3. .operation:返回底层 ir.Operation 对象,可做更底层操作

以 AddFOp 为例,ODS 自动生成的完整代码如下(位于 _arith_ops_gen.py):

@_ods_cext.register_operation(_Dialect)
class AddFOp(_ods_ir.OpView):
    OPERATION_NAME = "arith.addf"
    _ODS_REGIONS = (0, True)

    def __init__(self, lhs, rhs, *, fastmath=None, results=None, loc=None, ip=None):
        operands = []
        attributes = {}
        regions = None
        operands.append(lhs)
        operands.append(rhs)
        _ods_context = _ods_get_default_loc_context(loc)
        if fastmath is not None:
            attributes["fastmath"] = ...
        if results is None:
            results = [operands[0].type] * 1    # 结果类型 = 操作数类型(f32)
        _ods_successors = None
        super().__init__(
            self.OPERATION_NAME, self._ODS_REGIONS,
            self._ODS_OPERAND_SEGMENTS, self._ODS_RESULT_SEGMENTS,
            attributes=attributes, results=results, operands=operands,
            successors=_ods_successors, regions=regions, loc=loc, ip=ip,
        )

    @builtins.property
    def lhs(self) -> _ods_ir.Value:
        return self.operation.operands[0]

    @builtins.property
    def rhs(self) -> _ods_ir.Value:
        return self.operation.operands[1]

    @builtins.property
    def result(self) -> _ods_ir.OpResult:
        return self.operation.results[0]

可以看到,AddFOp.__init__ 的工作就是:收集参数 --> 组装 operands/attributes/results --> 调用 super().__init__().所有 Fly 方言 Op(LogicalDivideOp,MakeLayoutOp 等)的生成模式完全一致.

4.6 第四层:OpView.__init__() -- 真正插入 IR 的地方

OpView.__init__ 是 MLIR Python C++ 绑定(nanobind)的实现,核心逻辑等价于:

// 伪代码 - MLIR Python C++ 绑定内部
OpView::__init__(name, regions, ..., operands, loc, ip) {
    // 1. 确定 Location
    if (loc == None)
        loc = Location.current();   // 读取 Python 线程局部状态

    // 2. 确定 InsertionPoint
    if (ip == None)
        ip = InsertionPoint.current();  // 读取 Python 线程局部状态

    // 3. 创建 MLIR Operation
    OperationState state(loc, name);
    state.addOperands(operands);
    state.addTypes(result_types);
    state.addAttributes(attributes);

    // 4. 创建 Op 并插入到 Block
    Operation *op = Operation::create(state);
    ip.block.push_back(op);             // 追加到 Block 末尾

    // 5. InsertionPoint 自动前进到 op 之后
}

两个关键的线程局部状态:

状态类型设置方式读取时机
InsertionPointir.InsertionPointwith ir.InsertionPoint(block):OpView.__init__ 中 ip=None 时
Locationir.Locationwith loc:OpView.__init__ 中 loc=None 时
InsertionPoint 的工作机制

InsertionPoint 维护一个指向 Block 中某个位置的游标.每次 OpView.__init__ 插入一个 Op 后,游标自动前进到该 Op 之后,保证 Op 按执行顺序排列.

InsertionPoint 在 kernel_function.py 中设置:

with ir.InsertionPoint(entry_block), kernel_loc:
    # ... construct_from_ir_values ...
    self._func(**dsl_args)  # 所有 Op 都被插入到 entry_block 中

4.7 配合 traced_op 的 6 个接口体系

在列出具体接口之前,先回答一个根本问题:为什么需要 ir.Value <-> DSL 对象的互相转换?

MLIR 世界和 DSL 世界说的是两种"语言"----MLIR 只认识 ir.Value(SSA 值),而 DSL 对象需要方法调用(如 .shape,.stride).两个世界的"边界"就是下面这些转换接口:

DSL 世界
Tensor,Layout,Int32 ...
[Y] 可以调用 .shape,.stride
[Y] 可以参与 DSL 运算
[N] 不能直接传给 MLIR Op
__extract__
-->
<--
__construct__
MLIR 世界
ir.Value,ir.OpResult
[Y] 可以作为 Op 的 operand/result
[Y] 可以插入到 Block 中
[N] 无法调用 DSL 方法

转换发生在 4 个关键场景:

场景方向接口触发时机
traced_op 解包参数 DSL --> ir.Value __extract_to_ir_values__ 每个 fx.xxx() 调用时,_flatten_args 解包 DSL 对象
Kernel 参数重建 ir.Value --> DSL __construct_from_ir_values__ gpu.func 的 Block 参数需重建为 DSL 对象才能传入函数体
Kernel Launch 解包 DSL --> ir.Value __extract_to_ir_values__ KernelLauncher.launch() 解包参数传给 gpu.launch_func
OpResult 自动转回 ir.Value --> DSL register_value_caster Op 返回 ir.OpResult 时自动转为 DSL 对象,用户可链式调用
traced_op 的接口配合全景图
1. ir.InsertionPoint -- IR 插入位置
  • 设置:with ir.InsertionPoint(block):,在 kernel_function / jit_function 中
  • 读取:OpView.__init__ 中 ip=None 时读取 ir.InsertionPoint.current
  • 效果:新 Op 被插入到 InsertionPoint 指向的 Block 末尾,之后游标自动前进
2. ir.Location -- 调试位置
  • 设置方式 1:with loc:(traced_op 使用)
  • 设置方式 2:显式传递 fly.logical_divide(..., loc=loc)
  • 效果:IR 中每个 Op 都带有 loc("源码" at "文件:行:列") 信息
3. __extract_to_ir_values__ -- DSL 对象 --> ir.Value(解包)
  • BuiltinDslType(Tensor/Layout/IntTuple/CopyAtom 等):return [self](自身就是 ir.Value)
  • Numeric(Int32/Float32 等):return [self.ir_value()](需物化为 ir.Value,可能触发 arith.constant)
  • 触发场景:traced_op 的 _flatten_args 和 KernelLauncher 的 extract_to_ir_values
  • 为什么需要:MLIR Op 的 __init__ 只接受 ir.Value 作为 operand,不接受 DSL 对象
4. __construct_from_ir_values__ -- ir.Value --> DSL 对象(重建)
  • Kernel 参数重建:gpu.func 的 Block 参数是 ir.Value,需重建为 DSL 对象
  • Host 函数参数重建:JIT 函数的 ir.Value 参数需重建为 DSL 对象
  • 分派入口:protocol.construct_from_ir_values;典型实现:BuiltinDslType.__construct_from_ir_values__ --> cls(values[0])
  • 为什么需要:步骤 1 已改写好的函数体期望接收 DSL 对象(要调用 .shape 等方法),而非裸 ir.Value
5. __get_ir_types__ -- 获取 MLIR 类型信息
  • 分派入口:protocol.get_ir_types;各类型的 __get_ir_types__ 返回 MLIR 类型列表
  • 用于创建 gpu.func / func.func 时推导参数类型
  • Tensor.__get_ir_types__ --> [self.type]
  • Int32.__get_ir_types__ --> [i32]
6. ir.register_value_caster -- 自动类型转换
  • 当 MLIR Op 返回 ir.OpResult 时,根据类型自动转换为 DSL 对象
  • 例如:@ir.register_value_caster(TensorType.static_typeid, replace=True)
  • 效果:用户代码可直接调用 DSL 方法(如 layout.shape),无需手动包装
  • 为什么需要:Op 返回 ir.OpResult,但用户代码需要继续链式调用 DSL 方法(如 fx.slice(ta, ...))

5. 端到端示例:vector_add_kernel 的完整编译路径

让我们跟踪 vector_add_kernel 中一行代码的完整执行路径:

ta = fx.logical_divide(a, fx.make_layout(block_dim, 1))

5.1 阶段一:AST 改写(装饰时)

由于 vector_add_kernel 和 vector_add 都没有控制流,5 个 Transformer 都不修改 AST.步骤 1 改写后的函数体与原始代码完全相同.

5.2 阶段二:Kernel 发射(调用时)

当 vector_add_kernel(a, c, block_dim) 在 JIT 函数内被调用时:

  1. KernelFunction.__call__ --> _emit_kernel()
  2. 创建 gpu.func @vector_add_kernel_0,参数类型为 [TensorType, TensorType]
  3. 创建 entry_block,设置 ir.InsertionPoint(entry_block)
  4. 将 Block 参数重建为 DSL 对象:a = Tensor(%arg0), c = Tensor(%arg1)
  5. block_dim 是 Constexpr,直接传入值 64
  6. 调用 self._func(a=Tensor(...), c=Tensor(...), block_dim=64)
  7. 返回 KernelLauncher 对象(kernel 已定义但尚未启动)

5.3 阶段三:Tracing(函数体执行)

Tracing 的本质:执行步骤 1 已改写好的 Python 函数体,每遇到一个 fx.xxx() 调用,就实时插入一条 MLIR Op.函数体执行完毕时,gpu.func 的 body 也被填满了.

Tracing 前后对比(简化示例)

假设 kernel 函数体只有两行:

Tracing 前(Python 代码)
def vector_add_kernel(a, c, block_dim):
    ta = fx.logical_divide(a, fx.make_layout(block_dim, 1))
    tc = fx.logical_divide(c, fx.make_layout(block_dim, 1))
Tracing 后(gpu.func 内的 IR)
gpu.func @vector_add_kernel_0(%arg0, %arg1) {
  %0 = fly.make_layout(...)       loc("fx.make_layout(...)")
  %1 = fly.logical_divide(%arg0, %0) loc("ta = fx.logical_divide(...)")
  %2 = fly.make_layout(...)       loc("fx.make_layout(...)")
  %3 = fly.logical_divide(%arg1, %2) loc("tc = fx.logical_divide(...)")
  gpu.return
}

对应关系:每一行 fx.xxx() 执行时,traced_op 解包参数 --> OpView.__init__ 读取当前 InsertionPoint --> 创建 Op 并插入 Block --> InsertionPoint 自动前进到下一条 Op 之后.Python 变量(ta,tc)持有的是 OpResult 包装后的 DSL 对象,后续调用可以继续使用.

fx.logical_divide(a, fx.make_layout(block_dim, 1)) 的完整执行过程:

1. 先执行 fx.make_layout(block_dim, 1) | --> traced_op 包装器 | --> _caller_location() --> loc("fx.make_layout(block_dim, 1)" at "01-vectorAdd.py":28:28) | --> _flatten_args: block_dim=64 (int, 原样), 1 (int, 原样) | --> 进入 make_layout 函数体 | --> fly.make_layout(...) --> MakeLayoutOp.__init__() | --> OpView.__init__ 读取 InsertionPoint.current --> 插入 Op | --> 返回 Layout(%0) | 2. 执行 fx.logical_divide(a, layout_result) | --> traced_op 包装器 | --> _caller_location() --> loc("ta = fx.logical_divide(...)" at "01-vectorAdd.py":28:4) | --> _flatten_args: | a: Tensor --> __extract_to_ir_values__ --> [ir.Value(%arg0)] | layout_result: Layout --> __extract_to_ir_values__ --> [ir.Value(%0)] | --> with loc: --> 设置 Location 上下文 | --> 进入 logical_divide 函数体 | --> fly.logical_divide(%arg0, %0, loc=loc, ip=None) | --> LogicalDivideOp.__init__(layout=%arg0, divisor=%0, loc=loc, ip=None) | --> OpView.__init__: | loc 已提供(来自 traced_op) | ip=None --> 读取 ir.InsertionPoint.current(指向 entry_block) | 创建 fly.logical_divide Operation | 插入到 entry_block 末尾 | InsertionPoint 自动前进 | --> 返回 OpResult --> register_value_caster 自动转为 Tensor | --> ta = Tensor(%1)

5.4 阶段四:Kernel 启动(.launch() 调用时)

Tracing 完成后,KernelFunction.__call__ 返回 KernelLauncher.用户调用 .launch() 时,在 host 函数(func.func)内发射 gpu.launch_func:

# 用户代码
vector_add_kernel(a, c, block_dim).launch(grid=(grid_x,1,1), block=[block_dim,1,1], stream=stream)

# 生成的 IR(在 func.func 内部)
func.func @vector_add(...) {
  ...
  gpu.launch_func @kernels::vector_add_kernel_0
    grid_size = (%grid_x, %grid_y, %grid_z)
    block_size = (%block_x, %block_y, %block_z)
    args(%arg0, %arg1)
  ...
}

func.func @vector_add(...) {

这就是第一个 IR----由 Python tracing 直接生成.gpu.module 内是 kernel 定义(Fly 方言 IR),func.func 内是 host 代码(含 gpu.launch_func).后续 Pass Pipeline 会对其进行一系列变换(canonicalize,SSA 提升,lowering 等),最终生成硬件可执行的代码.

阶段发生时机核心机制
AST 改写装饰时(@flyc.kernel / @flyc.jit)5 个 Transformer 将 Python 控制流(if/for/while/and/or/not)改写为 scf_*_dispatch / dsl_*_ 调度函数;注入全局函数使改写后的代码可运行
Kernel 发射kernel 被调用时创建 gpu.func + entry_block;设置 InsertionPoint;将 Block 参数重建为 DSL 对象;执行步骤 1 已改写好的函数体触发 tracing;返回 KernelLauncher
Kernel 启动用户调用 .launch() 时KernelLauncher.launch() 发射 gpu.launch_func;解包参数,转换 grid/block 维度;在 func.func 内引用已定义的 gpu.func
traced_op 搭台每个 fx.xxx() 调用时自动推断 Location + 解包 DSL 对象为 ir.Value + 设置 Location 上下文
OpView 插入 IROp 构造器执行时读取 InsertionPoint.current 确定插入位置;读取 Location.current 确定调试位置;创建 Operation 并插入 Block
Pass Pipelinetracing 完成后(MlirCompiler.compile)对 origin IR 运行 canonicalize、layout-lowering、GPU lowering 等,生成可执行代码
6 个配合接口贯穿全流程InsertionPoint,Location,__extract_to_ir_values__,__construct_from_ir_values__,__get_ir_types__,register_value_caster

一句话总结:AST 改写让 Python 控制流变成 scf 调度函数,Kernel 发射搭建 gpu.func 舞台,traced_op 搭台(Location + 解包),OpView 唱戏(真正插入 IR),KernelLauncher.launch() 发射 gpu.launch_func 启动 kernel,MlirCompiler.compile 运行 Pass 生成可执行代码,6 个接口贯穿始终.
