1. 整体架构:MmaAtom 四层模型与数据流

核心思想 — Type-erased Wrapper + Interface Dispatch: Fly Dialect 不直接编码 MFMA/WMMA 等硬件张量核心指令.它只持有一个 !fly.mma_atom<InnerMmaOp> 包装类型.InnerMmaOp 是 Fly 通用类型(如 !fly.universal_fma<f32>)或后端 Dialect(fly_rocdl)定义的纯 MLIR 类型,实现了 Fly_MmaOpTypeInterface,通过 emitAtomCall / emitAtomCallSSA 把自己 lower 成 ROCDL intrinsic 或 LLVM FMA.

因此新增一种硬件 MMA 指令 = 在后端 Dialect 加一个 TypeDef + 实现 MmaOpTypeInterface,Fly Dialect 与上层 Python DSL 通常无需修改.

1.3 整体数据流(03-tiledMma 为例)

── Python DSL ── fx.rocdl.MFMA(16,16,4, f32) → MmaOp 类型 fx.make_mma_atom(mma_op) → fly.make_mma_atom fx.make_tiled_mma(atom, layout) → fly.make_tiled_mma fx.gemm(mma_atom, D, A, B, C) → fly.gemm (D=A×B+C) │ ▼ Pass: fly-convert-atom-call-to-ssa-form Fly IR (SSA) fly.mma_atom_call_ssa(%atom, %d?, %a, %b, %c) : vector<...> │ ▼ Pass: convert-fly-to-rocdl Lowering MmaAtomType::emitAtomCallSSA → MmaOpTypeInterface::emitAtomCallSSA ▼ Target IR UniversalFMA: llvm.fmul + llvm.fadd CDNA3 MFMA: rocdl.mfma_f32_16x16x4f32
与 CopyAtom 的对称关系

MmaAtom 与 CopyAtom 共用同一套 Atom 设计模式:包装类型 + Interface dispatch + StatefulOpTypeInterface + SSA Pass + FlyToROCDL lowering.主要差异:MMA 有 (M,N,K) 与 A/B/C 三套 TV layout;调用语义是 D = A×B + C 而非 src→dst 搬运;无 valBits 参数.

方法返回/行为谁消费

与 CopyOpTypeInterface 的对比

CopyOp 用 bit-level getThrBitLayoutSrc/Dst/Ref + valBits recast;MmaOp 直接提供 value-level 的 A/B/C TV layout,且多 getShapeMNK 与四套 val type.

2.2 Fly_StatefulOpTypeInterface — 何时需要

例如 CDNA4 MmaOpCDNA4_MFMAScaleType 带 scale_a / scale_b 运行时状态:

  • fly.make_mma_atom lower 为 getDefaultState() 返回的 LLVM struct
  • Python 通过 mma_atom.set_value("scale_a", v) → fly.atom.set_value → setAtomState
  • fx.gemm(mma_atom.set_value(...), d, a, b, c) 走 fly.gemm(operand 为 AnyType)

mma_atom_call vs gemm

fly.mma_atom_call 是一次原子级 MMA,operand 为四个 MemRef/SSA fragment.fly.gemm 是高层 GEMM,可接受 mma_atom 或 tiled_mma,并可选 traversal layout;fly-layout-lowering 可将其展开为循环 + 多次 mma_atom_call.简单 kernel(如 03-tiledMma.py)通常直接 fx.gemm(mma_atom, ...).

6. TiledMma 与 Layout 推导

T TiledMmaType
!fly.tiled_mma<atom, atomLayout, perm>  |  fly.make_tiled_mma

TiledMma 描述一个 block 内如何用多个 MmaAtom 覆盖更大的 GEMM tile,并驱动 A/B/C 的 partition 与 fragment 布局.

MmaAtom — 最小单元:一条 MFMA/WMMA 的 MNK 与 TV layout │ ▼ + atomLayout(如 (2,2,1)) + permutation(M,N,K) TiledMma — 平铺描述:tile_size_mnk、thr_layout_vmnk、tv_layout_*_tiled │ ▼ partition / make_fragment Register fragments — 每个 thread 持有的 A/B/C 子块 │ ▼ gemm / mma_atom_call — D = A×B + C

Layout 工具:include/flydsl/Dialect/Fly/Utils/TiledOpUtils.h 中的 layoutTiledMmaThrValView 等;Python 通过 TiledOpTraits.cpp 暴露 tile_size_mnk、thr_layout_vmnk、tiled_tv_layout_a/b/c.

Lowering 时 MakeTiledMmaOpLowering 透传 mmaAtom 操作数(FlyTypeConverter 将 TiledMmaType 映射为内部 MmaAtom 的转换后类型).

与 Copy 管线协作

典型 GEMM kernel 同时用 TiledMma 与 TiledCopy:make_tiled_copy_A/B/C(copy_atom, tiled_mma)(derived.py) 生成与 MMA operand layout 对齐的拷贝平铺,先 copy 数据到 fragment,再 fx.gemm.

7.1 编译阶段一览

阶段Pass / 组件对 MmaAtom 做什么
Tracing @traced_op 插入 make_mma_atom、gemm、mma.make_fragment 等
Layout fly-layout-lowering 展开 fly.gemm 为 mma_atom_call 循环(LayoutLowering.cpp)
SSA 提升 fly-convert-atom-call-to-ssa-form Register MemRef eligible 时:mma_atom_call → mma_atom_call_ssa
Vector SSA fly-promote-regmem-to-vectorssa 按 TV layout 将 register fragment 提升为 vector SSA
Backend convert-fly-to-rocdl MmaAtomCall*Lowering → emitAtomCall* → ROCDL MFMA/WMMA
P fly-convert-atom-call-to-ssa-form (MMA 路径)

与 CopyAtom 共用同一 Pass:对 MmaAtomCall,若 d/a/b/c 中任一 Register MemRef 满足 isEligibleToPromote,则先 ptr_load 为 SSA,再插入 mma_atom_call_ssa,最后对 d 端 ptr_store.

10.3 总结

要记住的 5 件事说明
1. 四层模型 MmaOp(指令) → MmaAtom(包装) → TiledMma(平铺) → mma_atom_call / gemm(执行)
2. Interface 是扩展点 所有 lower 逻辑在 MmaOpTypeInterface::emitAtomCall*;MmaAtomType 只转发
3. 语义 D=A×B+C 四个 operand d/a/b/c;SSA 路径返回累加结果 vector
4. 三套 TV layout A/B/C 各有 getThrValLayout*;驱动 partition 与 fragment 类型
5. Python 三层 MLIR 绑定 → primitive(@traced_op) → typing(MmaAtom/TiledMma 增强)

一句话:MmaAtom 是 Fly 方言对「矩阵乘指令」的类型擦除包装;具体生成哪条 MFMA/WMMA 完全由内嵌 MmaOp 的 Interface 实现决定;Python 侧通过 make_mma_atom 组装类型、通过 gemm / mma_atom_call 插入 IR,再经 SSA Pass 与 FlyToROCDL 落到 ROCDL intrinsic.
