1. 代码中 Layout 操作的含义

1.1 示例代码回顾

01-vectorAdd.py kernel 主体（与仓库示例一致）：

ta = fx.logical_divide(a, fx.make_layout(block_dim, 1))   # (1)
tc = fx.logical_divide(c, fx.make_layout(block_dim, 1))

ta = fx.slice(ta, (None, bid))                             # (2)
tc = fx.slice(tc, (None, bid))

ta = fx.logical_divide(ta, fx.make_layout(1, 1))           # (3)
tc = fx.logical_divide(tc, fx.make_layout(1, 1))

reg_ty = fx.MemRefType.get(fx.T.f32(), fx.LayoutType.get(1, 1), fx.AddressSpace.Register)
copy_atom = fx.make_copy_atom(fx.UniversalCopy32b(), fx.Float32)

ra = fx.memref_alloca(reg_ty, fx.make_layout(1, 1))        # (4)
rc = fx.memref_alloca(reg_ty, fx.make_layout(1, 1))

fx.copy_atom_call(copy_atom, fx.slice(ta, (None, tid)), ra) # (5) load

vc = fx.arith.addf(fx.memref_load_vec(ra), fx.memref_load_vec(ra))
fx.memref_store_vec(vc, rc)

fx.copy_atom_call(copy_atom, rc, fx.slice(tc, (None, tid))) # store

下文 Sec.1.3-Sec.1.6 重点讲解 (1)-(3) 的 Layout 映射；CopyAtom 与 SSA 见 Sec.2-Sec.5。

1.2 Layout 的核心概念：Shape + Stride

在 FlyDSL 中，Layout = (Shape, Stride)，它描述了逻辑坐标到物理内存偏移的映射关系：

Index = Sum(coord_i x stride_i)

例如 make_layout(64, 1) 表示：

  • Shape = 64，即有 64 个元素
  • Stride = 1，即相邻逻辑坐标在内存中偏移 1 个元素

这对应一个连续的一维数组访问模式。

1.3 logical_divide：用 Layout 切分 Tensor

logical_divide(tensor, divisor_layout) 是 FlyDSL 中最核心的 tiling 操作之一。

其语义是：将一个 tensor 按 divisor_layout 进行分块，产生分层 layout。layoutLogicalDivide 将 divisor 放在第 0 维、complement 放在第 1 维（见 LayoutUtils.h）。

在 vectorAdd 中的含义：

# (1) logical_divide(a, make_layout(block_dim=64, stride=1))

假设全局 tensor a 的大小为 N，用 layout (64, 1) 做 logical_divide：

  • 结果 layout 的 shape 变为 (64, N/64)，stride 变为 (1, 64)
  • 第 0 维 (64)：每个 block 内的元素索引
  • 第 1 维 (N/64)：block 索引（即 bid 切片所用维度）

这实际上是在说：把一维数组按 64 个元素一组切分，每组由一个 GPU block 处理。

读者注意：shape 维度顺序

直觉上常把 block 索引写成 shape 的第 0 维、块内 64 元素写成第 1 维，即 (N/64, 64):(64, 1)----这与 FlyDSL 实现不一致，也与 Sec.1.4 中 slice(ta, (None, bid)) 的用法矛盾（None 保留第 0 维、bid 切第 1 维，说明第 0 维才是块内元素）。

layoutLogicalDivide（include/flydsl/Dialect/Fly/Utils/LayoutUtils.h）构造结果 layout 时先 push divisor、再 push complement，因此 divisor 总是落在第 0 维：

rhsShapeElems.push_back(builder.getShape(divisorLayout));   // 第 0 维
rhsShapeElems.push_back(builder.getShape(complement));       // 第 1 维
rhsStrideElems.push_back(builder.getStride(divisorLayout));
rhsStrideElems.push_back(builder.getStride(complement));

对全局 layout N:1 除以 divisor 64:1，complement 的 shape 为 N/64、stride 为 64，故正确结果为 (64, N/64):(1, 64)：

  • 第 0 维：size=64, stride=1 -- 块内元素（slice(..., (None, bid)) 中 None 保留的这一维）
  • 第 1 维：size=N/64, stride=64 -- block 索引（bid 切片所用维度）

Sec.1.6 流程图中的 (64, N/64):(1, 64) 与上述约定一致；阅读 IR 或手写 layout 时请按实现顺序（divisor 在前）理解，而非按「外层 block、内层 tile」的口头习惯。

1.4 slice：按坐标切片

slice(tensor, coord) 在指定维度上选取一个坐标值，将那个维度"消除"。

# (2) slice(ta, (None, bid))
  • None 表示保留第 0 维（块内维度，size=64）
  • bid 表示在第 1 维（block 索引，size=N/64）选取当前 block

切片后，ta 从 (64, N/64) 变为 (64,)，即当前 block 负责的那 64 个元素。坐标含义与 Sec.1.3 维度顺序说明 一致。

1.5 第二次 logical_divide：为 thread 切分

# (3) logical_divide(ta, make_layout(1, 1))

这里 divisor 是 (1, 1)，即每个"块"只有 1 个元素。这实际上是在为后续按 thread 索引切片做准备----将每个元素独立出来，使得可以用 slice(ta, (None, tid)) 取出当前 thread 负责的那一个元素。

1.6 Layout 操作的整体流程图

全局 Tensor a (shape=N) | v logical_divide(a, (64, 1)) Tensor with layout (64, N/64):(1, 64) | v slice(..., (None, bid)) 当前 Block 的 Tensor (shape=64) | v logical_divide(..., (1, 1)) Tensor with layout (64, 1):(1, 1) | v slice(..., (None, tid)) 当前 Thread 的 Tensor (shape=1) <- 单个 f32 元素
核心思想
通过 logical_divide + slice 的组合，FlyDSL 用 Layout 代数精确描述了"全局数据 -> Block -> Thread"的层次化映射关系，而非依赖手动的 index 计算。

2. 什么是 Atom ---- 深入理解 CopyAtom

2.1 Atom 的概念

在 FlyDSL 中，Atom 是对硬件基本操作单元的抽象。它回答了一个核心问题：

核心问题
"一次原子操作中，哪些 thread 参与了操作，每个 thread 操作了哪些数据？"

FlyDSL 中有两种 Atom：

  • CopyAtom：描述数据搬运（load/store）的原子操作
  • MmaAtom：描述矩阵乘法（MMA）的原子操作

2.2 CopyAtom 的定义

从 FlyTypeDefs.td 中可以看到 CopyAtom 的核心接口：

CopyAtomType {
  copyOp    : Type       // 底层 CopyOp 类型
  valBits   : int32_t    // 每个值的位宽

  // 关键属性
  getThrLayout()       -> Attribute   // Thread 布局：哪些 thread 参与
  getThrValLayoutSrc() -> Attribute   // Thread-Value 布局（源端）
  getThrValLayoutDst() -> Attribute   // Thread-Value 布局（目标端）
  getThrValLayoutRef() -> Attribute   // Thread-Value 布局（参考端）

  // 关键方法
  emitAtomCall(...)     -> LogicalResult     // 发出 MemRef 形式的拷贝 IR
  emitAtomCallSSA(...)  -> FailureOr<Value>  // 发出 SSA 形式的拷贝 IR
}

2.3 Thread-Value (TV) Layout 详解

TV Layout 是 CopyAtom 最核心的概念，它用 (Thr, Val) 二元组描述：

  • Thr 维度：哪些 thread 参与这次原子操作
  • Val 维度：每个 thread 操作了哪些数据值

对于 UniversalCopy32b + Float32，其 TV Layout 为：

ThrLayout       = (1):(1)              // 1 个 thread
ThrValLayoutSrc = ((1, 1):(1, 1))       // 1 个 thread，1 个 f32 值
ThrValLayoutDst = ((1, 1):(1, 1))       // 同上

这意味着：每个 thread 独立搬运 32 bit（即一个 f32）的数据，没有跨 thread 协作。

2.4 make_copy_atom 的过程

Op fly.make_copy_atom
fx.UniversalCopy32b() + fx.Float32 -> fly.make_copy_atom

fx.make_copy_atom(copy_op, elem_type) 构造一个 CopyAtom----数据搬运的最小执行单元描述符.CopyOp 类型(!fly.universal_copy<32>) 由第一个参数 fx.UniversalCopy32b() 事先构造,并非 make_copy_atom 创建.

MakeCopyAtomOp 定义

def Fly_MakeCopyAtomOp : Fly_Op<"make_copy_atom", [Pure]> {
  let arguments = (ins I32Attr:$valBits);
  let results = (outs Fly_CopyAtom:$result);
  let assemblyFormat = "attr-dict `:` qualified(type($result))";
}

Python 两个参数 vs Op 一个 attribute

Python 侧 fx.make_copy_atom(copy_op, elem_type) 有两个参数,但 Fly IR 的 fly.make_copy_atom 在 TableGen 里只有 valBits 一个 attribute----第二个参数和 CopyOp 并没有丢失,而是拆到不同位置:

Python 参数Tracing 如何处理出现在 Fly IR 哪里
copy_op（如 UniversalCopy32b()） 传入 CopyAtomType.get(copy_op, valBits) 编码在 result type 的内嵌 CopyOp 里:!fly.copy_atom<!fly.universal_copy<32>, 32> 中的 !fly.universal_copy<32>
elem_type（如 Float32） 提取 valBits = 32 同时写入 result type 的第二个参数 , 32 和 op attribute {valBits = 32 : i32}

Fly_MakeCopyAtomOp 是 [Pure] 的"类型构造"op:CopyOp 种类由 result 的 !fly.copy_atom<copyOp, valBits> 完全确定,不需要再占一个 operand;attribute 里保留 valBits 是为 assembly/parser 与 verifier 方便.对应 Python 调用:

copy_atom_ty = CopyAtomType.get(copy_op=copy_op_type, val_bits=val_bits)
return fly.make_copy_atom(copy_atom_ty, val_bits=val_bits, ...)

第一个实参 copy_atom_ty 决定 op 的 result type(含 copyOp);第二个 val_bits 写入 attribute.

Tracing 三步

步骤位置说明
(1) python/flydsl/expr/primitive.py 从 elem_type(Float32) 提取 valBits = 32
(2) CopyAtomType.get(copy_op, valBits) 将 CopyOp 与 valBits 组合为 !fly.copy_atom<!fly.universal_copy<32>, 32>
(3) fly.make_copy_atom 插入 Pure op,产生 CopyAtom SSA 值;语义全部编码在 result type 中
Python DSL
copy_atom = fx.make_copy_atom(
    fx.UniversalCopy32b(), fx.Float32)
Tracing 生成的 Fly IR
%atom = fly.make_copy_atom {valBits = 32 : i32}
  : !fly.copy_atom<
      !fly.universal_copy<32>, 32>
CopyAtomType 类型参数

FlyTypeDefs.td 中 CopyAtomType 含 copyOp 与 valBits 两个参数,assembly 为 !fly.copy_atom<copyOp, valBits>.op 的 {valBits = ...} attribute 与类型中的 valBits 一致.

2.5 CopyAtom 的层次结构

CopyOpTypeInterface (接口)
  +-- CopyOpUniversalCopy<bitSize>     <- 通用 memcpy 语义
  +-- CopyOpUniversalAtomic<op, type>  <- 通用原子操作语义
  +-- (硬件特定实现)
       +-- CopyOpCDNA3BufferCopy       <- AMD CDNA3 Buffer 拷贝

  -> 封装了 thread 布局 + TV 布局 + emit 逻辑

3. CopyOpUniversalCopy 与 CopyAtom 的配合使用

3.1 CopyOpUniversalCopy 的语义

从 FlyUniversalOps.cpp 中可以看到：

Attribute CopyOpUniversalCopyType::getThrLayout() const {
    return FxLayout(FxC(1), FxC(1));  // 1 个 thread
}

Attribute CopyOpUniversalCopyType::getThrBitLayoutSrc() const {
    return FxLayout(FxShape(FxC(1), FxC(getBitSize())), FxStride(FxC(1), FxC(1)));
    // shape = (1 thread, bitSize bits), stride = (1, 1)
}

UniversalCopy 的特点：

  • 每个 thread 独立操作，无跨 thread 协作
  • 使用 memcpy 语义搬运数据
  • 是最通用的 CopyOp，不依赖任何硬件特定指令

3.2 从 CopyOp 到 CopyAtom：Layout 的 recast

CopyAtomType 在构建 TV Layout 时，会将 CopyOp 提供的 bit-level layout 通过 layoutRecast 转换为 value-level layout：

// CopyAtomType::getThrValLayoutSrc()
Attribute CopyAtomType::getThrValLayoutSrc() {
    auto copyOp = cast<CopyOpTypeInterface>(getCopyOp());
    LayoutBuilder<LayoutAttr> builder(getContext());
    // 将 bit layout 按 valBits 重新解释
    return layoutRecast(builder, cast<LayoutAttr>(copyOp.getThrBitLayoutSrc()),
                        1, getValBits());
}

对于 UniversalCopy32b + Float32：

  • Bit-level: (1, 32):(1, 1) -> 1 thread, 32 bits
  • Value-level (recast by 32): (1, 1):(1, 1) -> 1 thread, 1 个 f32 值

3.3 copy_atom_call 的执行流程

fx.copy_atom_call(copy_atom, fx.slice(ta, (None, tid)), ra)

这行代码的含义是：使用 copy_atom 描述的搬运模式，从 slice(ta, (None, tid))（全局内存中当前 thread 的数据）搬运到 ra（寄存器）。

Tracing 阶段 IR 保留 fly.copy_atom_call；在 convert-fly-to-ixdl（或 ROCDL 后端的 FlyToROCDL）中，由 CopyOp 的 emitAtomCall / emitAtomCallSSA 展开为具体 LLVM IR。对 UniversalCopy 有两条路径：

调用形式展开方式UniversalCopy 行为
copy_atom_call（MemRef operands） emitAtomCall 对 src/dst 指针生成一次 LLVM::MemcpyOp（bitSize/8 字节），不是按地址空间拆成单独的 Load + Store
copy_atom_call_ssa（SSA operands） emitAtomCallSSA 源为 MemRef -> LLVM::LoadOp；目标为 MemRef -> LLVM::StoreOp；Register->Register 直传

3.4 配合使用的完整数据流

CopyOpUniversalCopy<32> | 定义拷贝语义：1 thread 搬运 32 bit | ThrLayout = (1):(1) | ThrBitLayout = ((1, 32):(1, 1)) v CopyAtomType(UniversalCopy<32>, valBits=32) | ThrValLayoutSrc = ((1, 1):(1, 1)) <- recast | ThrValLayoutDst = ((1, 1):(1, 1)) <- recast | emitAtomCall -> MemRef 路径: LLVM::MemcpyOp | emitAtomCallSSA -> SSA 路径: Load / Store / 直传 v copy_atom_call(atom, src_memref, dst_memref) | convert-fly-to-ixdl 时 emitAtomCall -> memcpy

4. CopyOpUniversalCopyType 接口方法逐行详解

4.1 接口继承层次

CopyOpUniversalCopyType
  +-- Fly_MayStaticTypeInterface    <- 是否为纯静态类型
  |     +-- isStatic()
  |     +-- rebuildStaticValue(builder, loc, currentValue)
  |
  +-- Fly_CopyOpTypeInterface       <- CopyOp 核心接口
        +-- getThrLayout()
        +-- getThrBitLayoutSrc()
        +-- getThrBitLayoutDst()
        +-- getThrBitLayoutRef()
        +-- emitAtomCall(...)           x 2 (无 predicate / 有 predicate)
        +-- emitAtomCallSSA(...)        x 2 (无 predicate / 有 predicate)

4.2 宏定义前置知识：FxLayout / FxC / FxShape / FxStride / FxThr / FxVal

这些宏定义在 ThrValLayoutMacro.h.inc 中，是构建 Layout 的便捷工具：

宏展开结果含义
FxC(val)IntTupleAttr::getLeafStatic(ctx, val)创建一个叶子节点的静态 IntTuple，值为 val，可理解为标量常量
FxT(a, b)IntTupleAttr::get(ArrayAttr::get(ctx, {a, b}))创建一个二元组 IntTuple
FxShape(a, b)FxT_2(a, b)创建 shape 部分（二元组），参数可以是 FxC(...) 或 FxThr(...) / FxVal(...)
FxStride(a, b)FxT_2(a, b)创建 stride 部分（二元组），参数同上
FxThr(a, b, ...)FxT(FxC(a), FxC(b), ...)创建 Thread 维度的 shape/stride
FxVal(a, b, ...)FxT(FxC(a), FxC(b), ...)创建 Value 维度的 shape/stride
FxLayout(shape, stride)LayoutAttr::get(shape, stride)用 shape + stride 构造一个 LayoutAttr

关键理解：FxLayout(FxShape(A, B), FxStride(C, D)) 构造的 Layout 语义为：

Layout: shape = (A, B), stride = (C, D)

即一个二维 layout，第 0 维大小为 A、步长为 C，第 1 维大小为 B、步长为 D。

在 CopyAtom 的 TV Layout 语境中，第 0 维代表 Thread，第 1 维代表 Value。

4.3 逐接口方法详解

4.3.1 isStatic() -- 来自 MayStaticTypeInterface

bool CopyOpUniversalCopyType::isStatic() const { return true; }

含义：CopyOpUniversalCopyType 是纯静态类型，它的所有参数（只有 bitSize）都是编译期常量，不依赖任何运行时值。

对比：ROCDL 的 stateful CopyOp（如 CopyOpCDNA3BufferCopy）实现 StatefulOpTypeInterface（运行时状态如 soffset），不实现 MayStaticTypeInterface。因此包装它的 CopyAtomType::isStatic() 在 dyn_cast<MayStaticTypeInterface> 失败时返回 false（见 FlyTypeDefs.cpp），与 UniversalCopy 的 isStatic() == true 形成对比。

4.3.2 rebuildStaticValue(builder, loc, currentValue)

Value CopyOpUniversalCopyType::rebuildStaticValue(OpBuilder &builder, Location loc,
                                                  Value currentValue) const {
  if (currentValue && isa<MakeCopyAtomOp>(currentValue.getDefiningOp()))
    return nullptr;  // 已经是正常形式，无需重建
  return MakeCopyAtomOp::create(builder, loc,
                                CopyAtomType::get(*this, getBitSize()), getBitSize());
}

含义：将此类型重建为"正常形式"的 SSA 值。在 fly-rewrite-func-signature Pass 中，静态类型的参数会从函数签名中移除，然后在函数体内通过此方法重建。

  • 如果当前值已经由 MakeCopyAtomOp 产生 -> 返回 nullptr，表示无需重建
  • 否则 -> 创建一个 fly.make_copy_atom 操作

4.3.3 getThrLayout()

Attribute CopyOpUniversalCopyType::getThrLayout() const {
    return FxLayout(FxC(1), FxC(1));
}

展开：LayoutAttr::get(IntTupleAttr::getLeafStatic(ctx, 1), IntTupleAttr::getLeafStatic(ctx, 1))

含义：Thread Layout = (shape: 1, stride: 1)

这是一个一维 layout，只有 1 个 thread 参与，步长为 1。在 TV Layout 的语境中，这个 layout 描述的是"哪些 thread 参与了这次原子操作"。UniversalCopy 是单 thread 独立操作，所以 Thr 维度只有 1。

为什么 stride 是 1？

Layout 的映射公式：物理索引 = coord x stride。Stride = 1 表示逻辑坐标到物理索引是紧密排列的：coord 0->索引 0, coord 1->索引 1, ..., coord 15->索引 15。

对于 UniversalCopy，只有 1 个 thread，stride 的值实际上不影响结果（0x1 = 0x任何值 = 0），但 stride=1 是最自然的"连续排列"语义。

对比硬件特定实现：CDNA4 的 CopyOpCDNA4LdsReadTranspose：

Attribute CopyOpCDNA4LdsReadTransposeType::getThrLayout() const {
    return FxLayout(FxC(16), FxC(1));  // 16 个 thread 协作
}

这里 16 个 thread 一起参与一次 LDS 读取转置操作，stride=1 表示这 16 个 thread 的 ID 是连续排列的（0, 1, 2, ..., 15），这由硬件指令要求连续 lane 的语义决定。

4.3.4 getThrBitLayoutSrc()

Attribute CopyOpUniversalCopyType::getThrBitLayoutSrc() const {
    return FxLayout(FxShape(FxC(1), FxC(getBitSize())), FxStride(FxC(1), FxC(1)));
}

展开（以 UniversalCopy32b 为例，getBitSize() = 32）：

LayoutAttr::get(
    IntTupleAttr::get(ArrayAttr::get(ctx, {
        IntTupleAttr::getLeafStatic(ctx, 1),     // Thr shape = 1
        IntTupleAttr::getLeafStatic(ctx, 32)     // Val shape = 32 (bits)
    })),
    IntTupleAttr::get(ArrayAttr::get(ctx, {
        IntTupleAttr::getLeafStatic(ctx, 1),     // Thr stride = 1
        IntTupleAttr::getLeafStatic(ctx, 1)      // Val stride = 1
    }))
)

含义：Thread-Value Bit Layout（源端）= shape: (1, 32), stride: (1, 1)

维度ShapeStride含义
第 0 维 (Thread)111 个 thread，thread 索引步长为 1
第 1 维 (Value)32132 bit 数据，bit 之间连续（步长 1）

这描述的是：1 个 thread 从源端读取 32 bit 连续数据。

注意：这里的 Value 维度单位是 bit，不是"值"的个数。后续 CopyAtomType 会通过 layoutRecast 将 bit-level layout 转换为 value-level layout。

4.3.5 getThrBitLayoutDst()

Attribute CopyOpUniversalCopyType::getThrBitLayoutDst() const {
    return FxLayout(FxShape(FxC(1), FxC(getBitSize())), FxStride(FxC(1), FxC(1)));
}

与 getThrBitLayoutSrc() 完全相同。1 个 thread 写入 32 bit 连续数据。

对比硬件特定实现：CDNA4 的 LdsReadTranspose 中，源端和目标端的 Bit Layout 不同（因为转置操作改变了数据排列）：

// 源端：16 thread 各读 64 bit，按行排列
getThrBitLayoutSrc() = FxLayout(FxShape(FxC(16), FxC(64)), FxStride(FxC(64), FxC(1)));
// 目标端：16 thread 各写 64 bit，但经过转置，stride 发生了变化
getThrBitLayoutDst() = FxLayout(FxShape(FxC(16), FxVal(4, 16)), FxStride(FxC(4), FxVal(1, 64)));

4.3.6 getThrBitLayoutRef()

Attribute CopyOpUniversalCopyType::getThrBitLayoutRef() const {
    return FxLayout(FxShape(FxC(1), FxC(getBitSize())), FxStride(FxC(1), FxC(1)));
}

与 Src/Dst 完全相同。

什么是 "Ref" 端？

Ref（Reference）Layout 是 TiledCopy 中用于计算"从 Atom 的参考视角到源端/目标端的映射"的基准：

// 从 Ref 到 Src 的映射
auto atomLayoutRef = cast<LayoutAttr>(copyAtom.getThrValLayoutRef());
auto atomLayoutSrc = cast<LayoutAttr>(copyAtom.getThrValLayoutSrc());
LayoutAttr refInv = layoutRightInverse(attrBuilder, atomLayoutRef);
LayoutAttr ref2src = layoutComposition(attrBuilder, refInv, atomLayoutSrc);

// 从 Ref 到 Dst 的映射
auto atomLayoutDst = cast<LayoutAttr>(copyAtom.getThrValLayoutDst());
LayoutAttr ref2dst = layoutComposition(attrBuilder, refInv, atomLayoutDst);

即：ref2src = right_inverse(Ref) . Src，ref2dst = right_inverse(Ref) . Dst。

对于 UniversalCopy，因为 Src = Dst = Ref，所以 ref2src 和 ref2dst 都是恒等映射。

为什么需要 Ref？
当源端和目标端的数据排列不同时（如转置操作），Ref 提供了一个"标准参考坐标系"，使得可以通过 Layout 代数精确描述源端到目标端的坐标变换。

4.3.7 emitAtomCall(...) -- 无 predicate 版本

LogicalResult CopyOpUniversalCopyType::emitAtomCall(
    OpBuilder &builder, Location loc,
    Type copyAtomTyArg, Type srcMemTyArg, Type dstMemTyArg,
    Value atomVal, Value src, Value dst) const
{
  auto srcMemTy = cast<fly::MemRefType>(srcMemTyArg);
  auto dstMemTy = cast<fly::MemRefType>(dstMemTyArg);

  if (!isa<LLVM::LLVMPointerType>(src.getType()) ||
      !isa<LLVM::LLVMPointerType>(dst.getType()))
    return failure();

  int32_t copyBytes = getBitSize() / 8;
  Value srcPtr = applySwizzleOnPtr(builder, loc, src, srcMemTy.getSwizzle());
  Value dstPtr = applySwizzleOnPtr(builder, loc, dst, dstMemTy.getSwizzle());
  Value len = arith::ConstantIntOp::create(builder, loc, copyBytes, 32);
  LLVM::MemcpyOp::create(builder, loc, dstPtr, srcPtr, len, false);

  return success();
}

含义：在 MemRef 形式下，生成 LLVM::MemcpyOp 来搬运 bitSize / 8 字节的数据。

参数类型含义
copyAtomTyArgTypeCopyAtom 的类型
srcMemTyArgType源端 MemRef 类型
dstMemTyArgType目标端 MemRef 类型
atomValValueAtom 的 SSA 值（UniversalCopy 不使用，因为无状态）
srcValue源端指针
dstValue目标端指针

关键点：UniversalCopy 的 MemRef 形式 lowering 就是简单的 memcpy，没有任何硬件加速。

4.3.8 emitAtomCall(...) -- 有 predicate 版本

LogicalResult CopyOpUniversalCopyType::emitAtomCall(
    OpBuilder &builder, Location loc,
    Type copyAtomTyArg, Type srcMemTyArg, Type dstMemTyArg, Type predMemTyArg,
    Value atomVal, Value src, Value dst, Value pred) const
{
  auto predMemTy = cast<fly::MemRefType>(predMemTyArg);
  Value predVal = LLVM::LoadOp::create(builder, loc, predMemTy.getElemTy(), pred);
  auto ifOp = scf::IfOp::create(builder, loc, TypeRange{}, predVal, false);
  builder.setInsertionPointToStart(&ifOp.getThenRegion().front());
  return emitAtomCall(builder, loc, copyAtomTyArg, srcMemTyArg, dstMemTyArg, atomVal, src, dst);
}

含义：带 predicate 的版本 = 加载 predicate 值 -> scf.if 包裹无 predicate 版本。

4.3.9 emitAtomCallSSA(...) -- 无 predicate 版本

FailureOr<Value> CopyOpUniversalCopyType::emitAtomCallSSA(
    OpBuilder &builder, Location loc,
    Type resultTy, Type copyAtomTyArg, Type srcTyArg, Type dstTyArg,
    Value atomVal, Value src, Value dst) const
{
  Value result;
  if (isa<fly::MemRefType>(srcTyArg)) {
    // 源端是内存 -> 生成 Load
    auto srcMemTy = cast<fly::MemRefType>(srcTyArg);
    Type loadTy = resultTy ? resultTy : builder.getIntegerType(getBitSize());
    Value srcPtr = applySwizzleOnPtr(builder, loc, src, srcMemTy.getSwizzle());
    result = LLVM::LoadOp::create(builder, loc, loadTy, srcPtr);
  } else {
    // 源端是寄存器 SSA 值 -> 直接使用
    result = src;
  }

  if (!resultTy) {
    // 目标端是内存 -> 生成 Store
    auto dstMemTy = cast<fly::MemRefType>(dstTyArg);
    Value dstPtr = applySwizzleOnPtr(builder, loc, dst, dstMemTy.getSwizzle());
    LLVM::StoreOp::create(builder, loc, result, dstPtr);
  }
  return result;
}

含义：SSA 形式的拷贝，根据源端/目标端是内存还是寄存器，生成不同的 IR：

源端目标端生成的 IR返回值
MemRef (内存)Register (SSA)Load加载的值
Register (SSA)MemRef (内存)StoreValue()
Register (SSA)Register (SSA)直接传递src

关键参数：resultTy 如果非空，说明目标端是 SSA，需要返回结果值；如果为空，说明目标端是内存，只需 store。

4.3.10 emitAtomCallSSA(...) -- 有 predicate 版本

FailureOr<Value> CopyOpUniversalCopyType::emitAtomCallSSA(
    OpBuilder &builder, Location loc,
    Type resultTy, ..., Value pred) const
{
  if (resultTy) {
    // 目标端是 SSA -> 需要 scf.if 返回值
    auto ifOp = scf::IfOp::create(builder, loc, resultTy, pred, true);
    // then: 执行拷贝，yield 结果
    auto result = emitAtomCallSSA(builder, loc, ...);
    scf::YieldOp::create(builder, loc, *result);
    // else: 不拷贝，yield 旧的目标值
    scf::YieldOp::create(builder, loc, dst);
    return ifOp.getResult(0);
  }
  // 目标端是内存 -> 只需 then 分支
  auto ifOp = scf::IfOp::create(builder, loc, TypeRange{}, pred, false);
  emitAtomCallSSA(builder, loc, ...);
  return Value();
}

含义：带 predicate 的 SSA 版本。关键区别：

  • 目标端是 SSA：需要 scf.if 的 then/else 两个分支，else 分支 yield 旧值 dst（因为不执行拷贝时，目标寄存器保持原值）
  • 目标端是内存：只需要 then 分支执行 store

5. SSA 相关 Pass 的具体作用

5.1 Iluvatar 后端的 Pass Pipeline

以下只列出与 Layout / SSA 相关的 Fly 方言 pass（完整 pipeline 在 python/flydsl/compiler/backends/iluvatar.py 中；convert-fly-to-ixdl 之后还有 convert-gpu-to-ixdl、gpu-to-llvm、gpu-module-to-binary 等）：

"fly-rewrite-func-signature",          # Pass 1
"fly-canonicalize",                     # Pass 2
"fly-layout-lowering",                  # Pass 3
"fly-int-swizzle-simplify",            # Pass 4
"canonicalize",                         # Pass 5
"fly-convert-atom-call-to-ssa-form",   # Pass 6 <- SSA 关键
"fly-promote-regmem-to-vectorssa",     # Pass 7 <- SSA 关键
"convert-fly-to-ixdl",                 # Pass 8
; ... 后续: gpu/ixdl/LLVM lowering ...

5.2 Pass 6：fly-convert-atom-call-to-ssa-form

目标：将 copy_atom_call（MemRef 形式）转换为 copy_atom_call_ssa（SSA 形式）。

为什么需要这个 Pass？

在原始 IR 中，寄存器数据通过 MemRef（内存引用）的方式访问。下面为 Tracing 刚生成的 Fly IR（仍使用 fly.memref_alloca）：

// Tracing IR（MemRef 形式）
%ra = fly.memref_alloca ... : !fly.memref<f32, (1,1), register>
fly.copy_atom_call %atom, %src, %ra
%val = fly.memref_load_vec %ra

这种形式的问题：

  • 寄存器数据被当作"内存"来处理，引入了不必要的 load/store
  • 阻碍了编译器对数据流的优化（如 CSE、死代码消除）
  • 与 SSA（Static Single Assignment）范式不一致

转换后的 SSA 形式（Pass 6 之后，示意）：

// SSA 形式（示意；完整 assembly 含 operandSegmentSizes 等）
%val = fly.copy_atom_call_ssa %atom, %src
  {operandSegmentSizes = array<i32: 1, 1, 0, 0>} : f32
// %val 可直接用于 arith，无需 memref_load_vec

转换逻辑：

  1. 遍历所有 CopyAtomCall 操作
  2. 判断 src/dst 是否为 Register 地址空间
  3. 如果 src 是 Register：先用 PtrLoadOp 将 MemRef 中的值加载为 SSA 值
  4. 如果 dst 是 Register：创建 CopyAtomCallSSA，结果类型为 RegMem2SSAType
  5. 用 PtrStoreOp 将 SSA 结果写回（后续 Pass 会消除）

5.3 Pass 7：fly-promote-regmem-to-vectorssa

目标：将所有 Register MemRef 彻底提升为 Vector SSA 值。

这是最关键的 SSA 提升 Pass，它做了以下事情：

  1. 识别 Register 分配：Pass 3 fly-layout-lowering 已将 memref_alloca lower 为 fly.make_ptr + fly.make_view；本 pass 在这些 register 指针上工作
  2. 计算 SSA 类型：将 MemRef<f32, (1,1), register> 转换为 f32（标量）或 vector<4xf32>（向量）
  3. 重写整个函数体：
    • make_ptr / make_view（register）-> 消除，用 SSA 值替代
    • memref_load_vec / ptr_load -> 直接使用对应的 SSA 值
    • memref_store_vec / ptr_store -> 更新 SSA 值的定义
    • copy_atom_call_ssa -> 已经是 SSA 形式，保持
  4. 处理控制流：
    • scf.for：将 Register SSA 值作为 iter_args 传递
    • scf.if：在 then/else 分支中正确传播 SSA 值
    • scf.while：类似处理

5.4 IR 对比示例

同一 kernel 在不同 pass 阶段的 IR 形态不同；下面用示意 IR对比 Pass 6 前后（省略 layout/类型细节）。

Before Pass 6（layout lowering 之后、SSA 转换之前）

Tracing 的 memref_alloca 经 Pass 3 已变为 pointer + view：

%ra = fly.make_ptr ... : !fly.pointer<register, f32>
%ra_view = fly.make_view %ra, %layout : ...
fly.copy_atom_call %atom, %src, %ra_view
%val = fly.ptr_load %ra_view : f32
%result = arith.addf %val, %val : f32
%rc = fly.make_ptr ... : !fly.pointer<register, f32>
%rc_view = fly.make_view %rc, %layout : ...
fly.ptr_store %result, %rc_view
fly.copy_atom_call %atom, %rc_view, %dst

After Pass 6 + 7（SSA 提升后，示意）

%val = fly.copy_atom_call_ssa %atom, %src
  {operandSegmentSizes = array<i32: 1, 1, 0, 0>} : f32
%result = arith.addf %val, %val : f32
fly.copy_atom_call_ssa %atom, %result, %dst
  {operandSegmentSizes = array<i32: 1, 1, 1, 0>}
关键收益
寄存器数据不再通过指针间接访问，而是作为 SSA 值直接流动，使得后续优化 Pass（CSE、常量折叠、死代码消除）能够生效。

5.5 其他相关 Pass

Pass 1：fly-rewrite-func-signature

在函数边界处将 DSL 类型转换为 LLVM 兼容的类型：

  • 完全静态的 DSL 参数从函数签名中移除，在函数体内通过 fly.static 重建
  • DSL 类型（IntTuple, Layout, MemRef 等）被打包为 LLVM struct，只包含动态分量
  • scf.for/scf.if/scf.while 中的 DSL 类型也做同样转换

Pass 3：fly-layout-lowering

将高层 Layout 代数操作 lowering 为底层算术操作，并将 register 侧的 fly.memref_alloca lower 为 fly.make_ptr + fly.make_view：

  • fly.logical_divide -> 一系列 fly.crd2idx + arith 操作
  • fly.slice -> 坐标计算 + offset 叠加
  • fly.memref_alloca（register）-> fly.make_ptr + fly.make_view

6. Iluvatar 架构下的数据类型与 Layout 优化

6.1 Iluvatar 硬件的访问特性

在 Iluvatar 的硬件架构下，每个 thread 访问 32 bit（即一个 f32 或 2 个 f16）是最优的访问粒度。这与 CUDA 的 32-bit word 对齐类似。

对于 f32 数据类型，一个 f32 恰好就是 32 bit，因此 每个 thread 访问一个 f32 天然就是最优的，无需额外调整。

6.2 fp32 场景：天然对齐

当前 01-vectorAdd.py 使用的就是最优方案：

# fp32 版本 -- 天然最优
copy_atom = fx.make_copy_atom(fx.UniversalCopy32b(), fx.Float32)
reg_ty = fx.MemRefType.get(fx.T.f32(), fx.LayoutType.get(1, 1), fx.AddressSpace.Register)
# 每个 thread 搬运 1 个 f32 = 32 bit，完美对齐 Iluvatar 最优访问粒度

Tracing 生成的 Fly IR（make_copy_atom 细节见 Sec.2.4）：

%atom = fly.make_copy_atom {valBits = 32 : i32}
  : !fly.copy_atom<
      !fly.universal_copy<32>, 32>
%ra = fly.memref_alloca ... {layout = (1,1)}
  : !fly.memref<f32, register>
fly.copy_atom_call %atom, %src, %ra
; -> 每个 thread 搬运 32 bit = 1 个 f32
; -> Pass 6/7 SSA 提升后变成 f32 标量

6.3 fp16 场景：需要凑齐 32 bit

如果数据类型从 fp32 改为 fp16，每个 fp16 只有 16 bit，直接使用 UniversalCopy16b 会浪费一半带宽：

# [NG] 未优化的 fp16 版本
copy_atom = fx.make_copy_atom(fx.UniversalCopy16b(), fx.Float16)
reg_ty = fx.MemRefType.get(fx.T.f16(), fx.LayoutType.get(1, 1), fx.AddressSpace.Register)
# 每个 thread 只搬运 16 bit = 1 个 f16，带宽利用率仅 50%

优化方案：让每个 thread 一次搬运 2 个 fp16 值（共 32 bit），充分利用硬件带宽：

# [OK] 优化后的 fp16 版本
copy_atom = fx.make_copy_atom(fx.UniversalCopy32b(), fx.Float16)
# valBits=16, 但 CopyOp 的 bitSize=32
# -> 每个 thread 搬运 32 bit = 2 个 fp16 值
reg_ty = fx.MemRefType.get(fx.T.f16(), fx.LayoutType.get(2, 1), fx.AddressSpace.Register)
# Layout 从 (1, 1) 改为 (2, 1)：每个 thread 持有 2 个连续的 fp16 值

6.4 Layout 调整对 TV Layout 的影响

当 CopyAtom 的 bitSize=32 而 valBits=16 时，layoutRecast 会产生不同的 TV Layout：

UniversalCopy32b:
  ThrBitLayoutSrc = ((1, 32):(1, 1))    // bit-level: 1 thread, 32 bits

CopyAtomType(UniversalCopy32b, valBits=16):
  ThrValLayoutSrc = layoutRecast(bitLayout, 1, 16)
                   = ((1, 2):(1, 1))     // value-level: 1 thread, 2 个 f16 值

这意味着 RegMem2SSAType 会将寄存器 MemRef 转换为 vector<2xf16>，而非标量 f16。

6.5 Layout 修改对 logical_divide 和 slice 的影响

当每个 thread 处理多个元素时，logical_divide 的 tiler 也需要相应调整：

# 原始：每个 thread 处理 1 个元素
ta = fx.logical_divide(a, fx.make_layout(block_dim, 1))
ta = fx.slice(ta, (None, bid))
ta = fx.logical_divide(ta, fx.make_layout(1, 1))
# slice(ta, (None, tid)) -> 1 个元素

# 优化：每个 thread 处理 2 个 fp16 元素
elements_per_thread = 2
ta = fx.logical_divide(a, fx.make_layout(block_dim * elements_per_thread, 1))
ta = fx.slice(ta, (None, bid))
ta = fx.logical_divide(ta, fx.make_layout(elements_per_thread, 1))
# slice(ta, (None, tid)) -> 2 个连续的 fp16 元素

6.6 优化决策总结

场景数据类型CopyAtom bitSizeRegister LayoutSSA 类型带宽利用率
标准 fp32f3232(1, 1)f32100% [OK]
未优化 fp16f1616(1, 1)f1650% [NG]
优化 fp16f1632(2, 1)vector<2xf16>100% [OK]
优化原则
在 Iluvatar 架构下，确保每个 thread 的单次访问对齐到 32-bit 边界。对于 f32 数据类型，每个 thread 访问一个 f32 天然最优；对于 f16 等小于 32 bit 的数据类型，需要通过增大每个 thread 处理的元素数量来凑齐 32 bit。

总结
