1. 整体架构:FlyToROCDL Conversion Pass 概览

FlyToROCDLConversionPass 是 FlyDSL 编译器后端的核心 Pass 之一,负责将 Fly 方言(fly)的高层 Op lowering 到 ROCDL/LLVM.它使用 MLIR 的 DialectConversion 框架,通过 applyPartialConversion 逐步替换非法方言中的 Op.

fly_rocdl 也被标为 illegal,但 IR 中通常不出现独立的 fly_rocdl.* Op----Copy/MMA 的 fly_rocdl 类型信息编码在 !fly.copy_atom / !fly.mma_atom 里,在 copy_atom_call / mma_atom_call lowering 时由 emitAtomCall / emitAtomCallSSA 直接生成 rocdl.* 或 llvm.*,而非单独的 fly_rocdl pattern.

1.1 合法性规则

Pass 定义了以下合法性规则:

类别方言/Op状态
合法方言arith, scf, vector, gpu, func, llvm, rocdl保留
非法方言fly, fly_rocdl必须被转换
合法 Op(例外)StaticOp, MakeIntTupleOp, MakeLayoutOp, MakeComposedLayoutOp构造器保留
动态合法func.func, gpu.func函数签名必须已类型转换
动态合法gpu.launch_func内核操作数类型必须合法

2. FlyTypeConverter:类型转换基础设施

所有 Conversion Pattern 共享同一个 FlyTypeConverter,它定义了 Fly DSL 类型到 LLVM/ROCDL 类型的映射规则:

Fly DSL 类型LLVM/ROCDL 类型说明
FloatType (width < 16)IntegerType(同位宽)fp8/fp6/fp4 在 LLVM 中无对应浮点类型,退化为等位宽整数
VectorType (含 sub-byte float)VectorType(元素类型已转换)递归应用元素类型转换
fly::MemRefType (BufferDesc)BufferFatPtr 结构体类型Buffer Resource 描述符打包
fly::MemRefType (其他)LLVMPointerType裸指针
fly::PointerType (BufferDesc)BufferFatPtr 结构体类型同 MemRef 的 BufferDesc 路径
fly::PointerType (其他)LLVMPointerType裸指针
fly::CopyAtomType (stateful)getConvertedType()Atom 的运行时状态结构体
fly::CopyAtomType (non-stateful)LLVMStructType{}(空)无运行时数据
fly::MmaAtomType同 CopyAtomType 逻辑同上
fly::TiledCopyTypeconvertType(copyAtom)透传到内部 atom 类型
fly::TiledMmaTypeconvertType(mmaAtom)透传到内部 atom 类型
为什么 fp8/fp6/fp4 要转为整数?

LLVM IR 不存在 f8/f6/f4 这样的一等浮点类型.AMD GPU 硬件约定 sub-byte 浮点数据在内存和寄存器中以整数形式存储和搬运,MFMA intrinsic 在操作数位置期望整数类型,硬件内部将 bit pattern 解释为浮点进行计算.因此 lowering 时必须将 f8 -> i8,f6 -> i6,f4 -> i4.

3. 指针创建与内存分配类 Conversion

1 MakePtrOpLowering
fly.make_ptr -> llvm.alloca / llvm.addressof / BufferFatPtr

MakePtrOpLowering 是最复杂的 conversion 之一,根据指针的地址空间走三条完全不同的路径:

地址空间Lowering 方式用途
Register LLVM::AllocaOp 寄存器分配,后续被 PromoteRegMemToVectorSSA 提升为标量/vector SSA
Shared LLVM::GlobalOp + LLVM::AddressOfOp 静态 LDS 共享内存分配
BufferDesc ROCDL::MakeBufferRsrcOp + BufferFatPtr::pack GPU 全局显存 Buffer Resource 描述符构造

路径 1:Register -> llvm.alloca

Register 地址空间对应 GPU 的私有地址空间(AS=5).fly.make_ptr 的 dictAttrs 中必须包含 allocSize 属性,表示需要分配的元素数量.

Lowering 生成一个 llvm.alloca,分配 allocSize 个 elemTy 元素的连续空间:

  • allocSize == 1:分配单个元素,后续被提升为标量 SSA 值(如 f16)
  • allocSize > 1:分配多个元素,后续被提升为 vector SSA 值(如 vector<8xf16>)

alloca 只是中间表示,后续 PromoteRegMemToVectorSSA pass 会将其完全消除,直接使用寄存器.

Before (Fly 方言)
%ptr = fly.make_ptr {allocSize = 8}
  : !fly.pointer<f16, register>
After MakePtrOpLowering
%nElems = arith.constant 8 : i64
%ptr = llvm.alloca %nElems :
  !llvm.ptr<5>, f16

路径 2:Shared -> LDS Global

Shared 地址空间对应 GPU 的本地数据共享(LDS)内存(AS=3).每个 fly.make_ptr 会在模块中创建一个唯一的 LDS 全局变量 @__shared_alloc_<n>,然后取其地址.

dictAttrs 中必须包含 allocBytes(分配字节数)和 allocAlign(对齐字节数).

Before (Fly 方言)
%ptr = fly.make_ptr
  {allocBytes = 2048, allocAlign = 256}
  : !fly.pointer<f16, shared>
After MakePtrOpLowering
; 模块级新增全局变量(external linkage, dso_local):
llvm.mlir.global external @__shared_alloc_0
  {addr_space = 3 : i32, dso_local}
  = zeroinitializer : !llvm.array<2048 x i8>

; op 替换为取地址:
%ptr = llvm.addressof
  @__shared_alloc_0 : !llvm.ptr<3>

路径 3:BufferDesc -> BufferFatPtr

BufferDesc 地址空间对应 AMD GPU 的 Buffer Resource 机制.需要 4 个参数构造:

  • base:buffer 基地址
  • stride:元素步长
  • numRecords:记录数
  • flags:buffer 标志位

Lowering 先用 ROCDL::MakeBufferRsrcOp 将 4 个参数打包为 buffer 描述符,再用 BufferFatPtr::pack 将其与零偏移量打包为 BufferFatPtr 结构体 {ptr<8>, i32}.

Before (Fly 方言)
%ptr = fly.make_ptr(%base, %stride,
  %numRecords, %flags)
  : !fly.pointer<f32, buffer_desc>
After MakePtrOpLowering
; 1. 构造 buffer resource:
%rsrc = rocdl.make.buffer.rsrc
  %base, %stride,
  %numRecords, %flags
  : !llvm.ptr<8>

; 2. 打包为 BufferFatPtr:
%zero = arith.constant 0 : i32
%ptr = llvm.insertvalue %rsrc, 0
  : {!llvm.ptr<8>, i32}
%ptr = llvm.insertvalue %zero, 1
  : {!llvm.ptr<8>, i32}
2 GetDynSharedOpLowering
fly.get_dyn_shared -> llvm.addressof + llvm.getelementptr

行为:获取动态共享内存基址

GetDynSharedOpLowering 将 fly.get_dyn_shared lowering 为动态共享内存的基址指针.与 MakePtrOpLowering 的 Shared 路径不同,动态共享内存的大小在编译期未知,由 kernel launch 时指定.

核心逻辑:复用同一个全局变量

动态共享内存每个 kernel 只有一个逻辑区域.代码会先在模块中查找是否已存在符合条件的 LDS 全局变量:

  • 地址空间为 shared(AS=3)
  • 类型为 !llvm.array<0 x i8>(零长度数组,表示大小动态)
  • 对齐为 1024 字节

如果找到则复用,否则创建一个新的 @__dynamic_shared 全局变量.这保证了同一个 kernel 中多个 get_dyn_shared 调用返回的是同一个基址.

为什么需要 GEP?

全局变量的类型是 !llvm.array<0 x i8>,取地址得到指向该数组的 !llvm.ptr<3>.实现中对 addressof 结果再做一次单索引 GEP(gep i8, ptr, 0),使指针语义与后续 offset / recast 路径一致:

Before (Fly 方言)
%ptr = fly.get_dyn_shared
  : !fly.pointer<f16, shared>
After GetDynSharedOpLowering
; 模块级(仅首次创建, external + dso_local):
llvm.mlir.global external @__dynamic_shared
  {addr_space = 3 : i32, alignment = 1024, dso_local}
  = zeroinitializer
  : !llvm.array<0 x i8>

; op 替换:
%base = llvm.addressof
  @__dynamic_shared : !llvm.ptr<3>
%ptr = llvm.getelementptr %base[0]
  : (!llvm.ptr<3>, !llvm.array<0 x i8>), i8

4. 指针类型转换类 Conversion

3 IntToPtrOpLowering
fly.int_to_ptr -> llvm.inttoptr

行为:整数转指针,类型由 FlyTypeConverter 决定

将 fly.int_to_ptr 直接转换为 llvm.inttoptr.关键点在于结果类型:Fly DSL 的 !fly.pointer 类型通过 FlyTypeConverter 转换为对应的 !llvm.ptr<AS> 或 BufferFatPtr 结构体类型,转换逻辑与指针的地址空间相关.

Before (Fly 方言)
%ptr = fly.int_to_ptr %int_val
  : !fly.pointer<f16, shared>
After IntToPtrOpLowering
%ptr = llvm.inttoptr %int_val
  : !llvm.ptr<3>
4 PtrToIntOpLowering
fly.ptr_to_int -> llvm.ptrtoint

行为:指针转整数,与 IntToPtrOpLowering 对称

将 fly.ptr_to_int 直接转换为 llvm.ptrtoint.结果类型同样通过 FlyTypeConverter 转换.

Before (Fly 方言)
%int_val = fly.ptr_to_int %ptr
  : i64
After PtrToIntOpLowering
%int_val = llvm.ptrtoint %ptr
  : i64

5. 指针变换与遍历类 Conversion

5 ApplySwizzleOpLowering
fly.apply_swizzle -> 透传指针

行为:直接消除,透传原始指针

ApplySwizzleOpLowering 的实现极其简单:将 fly.apply_swizzle op 直接消除,用原始指针替换它.看起来好像 swizzle 被完全丢弃了,但实际上并非如此----swizzle 的效果被延迟到了 load/store 时才真正应用.

为什么可以这样做?

关键在于理解 Fly DSL 中 swizzle 的设计哲学:swizzle 是指针类型上的元数据,而不是指针值上的变换.

在 Fly DSL 中,fly.apply_swizzle 的作用是给一个 !fly.pointer<elemTy, swizzle=...> 类型的指针附加上 swizzle 信息.swizzle 信息编码在指针类型的 SwizzleAttr 中,而不是改变指针的数值.所以 lowering 时:

  • ApplySwizzleOp 本身不需要修改指针值,只需透传
  • swizzle 信息随指针类型一起传播,后续使用者(load/store)可以从原始 op 的指针类型中读取到

IR 变换示例

Before (Fly 方言)
%ptr = fly.make_ptr ... :
  !fly.pointer<f16, shared,
    swizzle=<base=4,mask=3,shift=0>>
%swizzled = fly.apply_swizzle %ptr :
  !fly.pointer<f16, shared,
    swizzle=<base=4,mask=3,shift=0>>
%val = fly.ptr_load %swizzled : vector<4xf16>
After (LLVM/ROCDL)
%ptr = llvm.alloca ... : !llvm.ptr<3>
; apply_swizzle 被消除,
; %ptr 直接透传到 load
; swizzle 在 ptr_load 时应用:
%addr = applySwizzleOnPtr(%ptr,
          base=4,mask=3,shift=0)
%val = llvm.load %addr :
         vector<4xf16>

Swizzle 真正生效的地方

swizzle 的实际计算发生在 load/store lowering 时:

普通地址空间的 load(PtrLoadOpLowering):

ptr = applySwizzleOnPtr(rewriter, loc, ptr, flyPtrTy.getSwizzle());
Value loaded = LLVM::LoadOp::create(rewriter, loc, loadTy, ptr);

普通地址空间的 store(PtrStoreOpLowering):

ptr = applySwizzleOnPtr(rewriter, loc, ptr, flyPtrTy.getSwizzle());
LLVM::StoreOp::create(rewriter, loc, value, ptr);

而 applySwizzleOnPtr(定义在 PointerUtils.cpp)才是真正做位运算的地方:

TypedValue<LLVM::LLVMPointerType> applySwizzleOnPtr(...) {
  if (swizzle.isTrivialSwizzle())
    return ptr;                                    // 无 swizzle,直接返回
  Value ptrInt = LLVM::PtrToIntOp::create(...);    // 指针转整数
  int64_t bitMaskValue = ((1 << mask) - 1) << (base + shift);
  Value bitMask = arith::ConstantIntOp::create(..., bitMaskValue);
  Value masked = arith::AndIOp::create(..., ptrInt, bitMask);  // 提取 swizzle 位
  Value shifted = arith::ShRUIOp::create(..., masked, shiftAmt); // 右移
  Value swizzled = arith::XOrIOp::create(..., ptrInt, shifted); // XOR 得到新地址
  return LLVM::IntToPtrOp::create(..., ptrTy, swizzled);        // 整数转回指针
}

这是一个 XOR swizzle:address ^= (address & mask) >> shift,用于 AMD GPU 共享内存的 bank conflict 消除.

设计思路总结

Fly DSL 层面: %ptr_with_swizzle = fly.apply_swizzle %ptr <- 标记类型,不改变值 %val = fly.ptr_load %ptr_with_swizzle <- 使用带 swizzle 信息的指针 Lowering 后: ApplySwizzleOp -> 直接透传指针值 <- swizzle 信息在类型中 PtrLoadOp -> applySwizzleOnPtr(ptr, swizzle) <- 此处才计算 swizzle 地址 + llvm.load
延迟应用设计的两个好处
  • 避免重复计算:如果同一个指针被 load 多次,swizzle 只需在 load 时计算一次.如果 apply_swizzle 就修改了指针值,后续每次 add_offset 等操作都需要额外处理 swizzle 的传播.
  • BufferDesc 路径天然兼容:对于 buffer resource 指针,swizzle 偏移是作为 RawPtrBufferLoadOp 的独立参数传入的,不需要修改指针值本身.把 swizzle 逻辑集中在 load/store 处理,可以统一两种地址空间的代码路径.

BufferDesc 地址空间的 swizzle 处理

对于 BufferDesc 地址空间,swizzle 不通过 applySwizzleOnPtr 处理,而是作为 BufferFatPtr 的 swizzleByteOffset 参数直接传入 buffer load/store intrinsic:

// PtrLoadOpLowering 中 BufferDesc 路径:
BufferFatPtr bp(flyPtrTy, ptr);
Value loaded = ROCDL::RawPtrBufferLoadOp::create(
    rewriter, loc, loadTy,
    bp.bufferRsrc(rewriter, loc),       // buffer 描述符
    bp.swizzleByteOffset(rewriter, loc), // swizzle 偏移(独立参数)
    zero, zero, noAttrs, noAttrs, noAttrs);

这进一步说明了为什么 ApplySwizzleOpLowering 只需透传:不同地址空间的 swizzle 处理方式完全不同,统一在 load/store 时根据地址空间分发才是最合理的.

对已有 swizzle 的指针再次 apply_swizzle:覆盖而非组合

ApplySwizzleOp 的类型推断逻辑(FlyOps.cpp)揭示了关键语义:

FLY_INFER_RETURN_TYPES(ApplySwizzleOp) {
  auto ptrTy = dyn_cast<PointerType>(operands[0].getType());
  auto swizzleTy = dyn_cast<SwizzleType>(operands[1].getType());
  int32_t valDiv = ptrTy.getValueDivisibility();
  int32_t newValDiv = utils::divisibilityApplySwizzle(valDiv, swizzleTy.getAttr());
  inferredReturnTypes.assign(
      {PointerType::get(ptrTy.getElemTy(), ptrTy.getAddressSpace(),
                        AlignAttr::get(ptrTy.getElemTy(), newValDiv),
                        swizzleTy.getAttr())});   // <- 新 swizzle 直接覆盖,不与旧 swizzle 组合
  return success();
}

结果指针类型中的 swizzle 属性直接设为新的 swizzleTy.getAttr(),而非与旧 swizzle 做组合.旧的 swizzle 被完全丢弃.

具体例子

%ptr1 = fly.make_ptr ... : !fly.pointer<f16, shared, swizzle=<base=4,mask=3,shift=0>>
%sw1  = fly.static : !fly.swizzle<base=4,mask=3,shift=0>
%ptr2 = fly.apply_swizzle %ptr1, %sw1
       -> swizzle = <base=4,mask=3,shift=0>   // 覆盖,但恰好相同

%sw2  = fly.static : !fly.swizzle<base=8,mask=2,shift=1>
%ptr3 = fly.apply_swizzle %ptr2, %sw2
       -> swizzle = <base=8,mask=2,shift=1>   // 旧的 <4,3,0> 被丢弃!

在 lowering 时,ApplySwizzleOpLowering 直接透传指针值,而 PtrLoadOpLowering/PtrStoreOpLowering 从指针类型中读取 swizzle 并在 load/store 时应用.所以最终只有最后一次 apply_swizzle 设置的 swizzle 会在 load/store 时生效.

这不是 bug,而是设计意图

Fly DSL 的 swizzle 语义是描述性的,而非变换性的:

  • apply_swizzle 不是"对指针做一次 swizzle 变换",而是"声明这个指针当前具有的 swizzle 模式"
  • 指针类型上的 swizzle 是一个当前状态标记,告诉后续 load/store 如何解释地址
  • 如果需要组合两个 swizzle,应该在类型层面预先计算好组合后的 swizzle,然后一次性 apply_swizzle

这也解释了为什么 ApplySwizzleOpLowering 可以放心地直接透传指针----因为 swizzle 只是类型元数据,不存在"累积变换"的语义.

6 RecastIterOpLowering
fly.recast_iter -> 透传指针

行为:直接消除,透传原始指针

RecastIterOpLowering 将 fly.recast_iter 直接消除,用源操作数替换结果.与 ApplySwizzleOpLowering 一样,这看起来像是丢弃了信息,但实际上 被改变的信息编码在指针类型中,而类型在 lowering 后会被抹平.

RecastIterOp 的本质:指针的 reinterpret_cast

从 op 定义可以看到:

def Fly_RecastIterOp : Fly_Op<"recast_iter", [Pure]> {
  let arguments = (ins Fly_Pointer:$src);
  let results = (outs Fly_Pointer:$result);
}

它接受一个 !fly.pointer<A, ...> 输入,产生一个 !fly.pointer<B, ...> 输出.运行时值完全不变,变的是指针类型中的 elemTy(元素类型).

Python 端的用法也印证了这一点:

# 把共享内存指针从默认类型"重铸"为 f32 指针
smem_f32 = recast_iter(fx.Float32, get_dyn_shared())

# 从指针 peek 一个值:先 recast 成目标类型,再 load
typed_ptr = recast_iter(cls, ptr)
return cls(ptr_load(typed_ptr, cls))

# 往指针 poke 一个值:先 recast,再 store
typed_ptr = recast_iter(cls, ptr)
ptr_store(coerced.ir_value(), typed_ptr)

recast_iter 就是 C++ 中的 reinterpret_cast----只改变指针的类型标签,不改变指针的地址值.

IR 变换示例

Before (Fly 方言)
%ptr_f16 = fly.make_ptr ... :
  !fly.pointer<f16, shared>
%ptr_f32 = fly.recast_iter %ptr_f16 :
  !fly.pointer<f32, shared>
%val = fly.ptr_load %ptr_f32 : f32
After RecastIterOpLowering
%ptr_f16 = fly.make_ptr ... :
  !fly.pointer<f16, shared>
; recast_iter 被消除,
; %ptr_f16 直接透传
%val = fly.ptr_load %ptr_f16 : f32
; 注意:ptr_load 的指针操作数类型
; 仍是 !fly.pointer<f16, shared>,
; 但 load 结果类型不受影响

经过 RecastIterOpLowering 后,fly.recast_iter 被消除,源指针 %ptr_f16 直接透传到 fly.ptr_load.后续 PtrLoadOpLowering 会进一步将 fly.ptr_load lowering 为 applySwizzleOnPtr + llvm.load,此时指针类型中的 elemTy 已不再影响 load 行为.

地址空间可以不同吗?

不可以. Python 端 recast_iter 构造结果类型时,显式复用了源指针的地址空间和对齐:

result_type = PointerType.get(result_type, src.memspace, src.alignment)

结果指针的 memspace(地址空间)和 alignment 与源指针完全相同,只有 elemTy(元素类型)被替换.这是合理的:recast_iter 是对同一块内存的类型重解释,不可能把 shared memory 的指针"重铸"成 global memory 的指针----那需要地址空间转换,而非类型重铸.

地址空间校验现状

目前程序上没有强制限制,也没有报错:

  • MLIR 层面:RecastIterOp 的 .td 定义没有声明 InferTypeOpInterface,也没有自定义 verifier.结果类型完全由调用者提供,MLIR 不会校验结果指针的地址空间是否与源指针一致.
  • Python API 层面:有两条路径----
    • 快捷路径(传 Numeric 类):PointerType.get(result_type, src.memspace, src.alignment) 自动复用源指针的地址空间,不可能出错.
    • 显式路径(传 PointerType):如 fp8_gemm_utils.py 和 preshuffle_gemm_v2.py 中的用法,用户自己构造 PointerType 传入,此时地址空间由用户指定,Python API 不会校验是否与源指针一致.

如果用户错误地给了不同地址空间,RecastIterOpLowering 仍然直接透传指针值,不会报错.但后续 lowering 时地址空间会被编码到 llvm.ptr<AS> 中,可能导致语义错误----比如把 shared memory 的地址当作 global memory 使用,运行时行为未定义.

TODO: 需要向社区提交 Patch

应为 RecastIterOp 添加 verifier,校验结果指针的地址空间与源指针一致.如果不一致,应在编译期报错而非静默通过.同时,Python API 的显式路径也应增加校验.

为什么 Lowering 可以直接透传

Lowering 后所有 !fly.pointer<elemTy, ...> 都变成了 !llvm.ptr<AS>,元素类型信息被剥离.所以 recast_iter 改变的那个 elemTy 在底层已经没有意义,直接透传指针值即可.

类型级操作的设计模式

RecastIterOp 和 ApplySwizzleOp 都属于同一类设计模式----类型级操作:在 Fly DSL 高层 IR 中用于类型推导和语义约束,但在 lowering 到 LLVM 后,类型差异被抹平,op 本身变成了 no-op,只需透传运行时值.

7 AddOffsetOpLowering
fly.add_offset -> BufferFatPtr.addOffset / llvm.getelementptr

AddOffsetOp 定义

def Fly_AddOffsetOp : Fly_Op<"add_offset", [Pure, ...]> {
  let arguments = (ins Fly_IteratorLikeType:$ptr, Fly_IntTuple:$offset);
  let results = (outs Fly_IteratorLikeType:$result);
}

注意 offset 参数的类型是 Fly_IntTuple----Fly DSL 中整数元组类型.IntTupleAttr 是递归嵌套结构,可以是标量(leaf)也可以是元组(non-leaf).add_offset 要求 offset 必须是 leaf IntTuple(即标量整数),否则 lowering 会失败.

行为:按地址空间分两条路径

地址空间Lowering 方式说明
BufferDesc BufferFatPtr::addOffset() 修改结构体中的 valOffset 字段,返回新的 BufferFatPtr
其他 LLVM::GEPOp 标准 LLVM getelementptr,按元素类型计算字节偏移

Offset 值的提取

offset 参数是 IntTupleType,但 lowering 需要一个具体的整数值.提取逻辑分两种情况:

  • 静态 offset(编译期已知):offsetAttr.extractIntFromLeaf() 得到 IntAttr,如果 isStatic() 为 true,直接用 arith.ConstantIntOp 生成常量.
  • 动态 offset(运行时值):offset 的定义 op 的第一个操作数就是实际的运行时整数值,直接取 defOp->getOperand(0).

IR 变换示例 (普通地址空间)

Before (Fly 方言)
%offset = fly.static 4 : !fly.int_tuple<4>
%new_ptr = fly.add_offset %ptr, %offset
  : !fly.pointer<f16, shared>
After AddOffsetOpLowering
%offset_val = arith.constant 4 : i32
%new_ptr = llvm.getelementptr %ptr[%offset_val]
  : !llvm.ptr<3>, f16
8 MakeViewOpLowering
fly.make_view -> 透传迭代器 / 消除 op

MakeViewOp 定义

def Fly_MakeViewOp : Fly_Op<"make_view", [Pure, ...]> {
  let arguments = (ins Fly_IteratorLikeType:$iter, Fly_NarrowLayoutType:$layout);
  let results = (outs Fly_TensorLikeType:$result);

  • iter 是 PointerType -> 结果是 MemRefType(内存引用)

两条路径

结果类型Lowering 方式原因
CoordTensorType eraseOp(直接删除) CoordTensor 是纯编译期信息,无运行时表示;且结果不应有使用者

  : vector<4xf16>
; (layout-lowering 通常已将 memref.load 展开为 ptr 路径)
After MakeViewOpLowering

  : !fly.pointer<f16, shared>

; make_view 被消除,
; 迭代器 %ptr 直接透传到后续 ptr.load
%val = fly.ptr.load %ptr
  : vector<4xf16>

6. 内存读写类 Conversion

9 PtrLoadOpLowering
fly.ptr_load -> applySwizzleOnPtr + llvm.load / RawPtrBufferLoad

整体行为:swizzle 在此处真正生效

与 ApplySwizzleOpLowering 和 RecastIterOpLowering 的"透传"不同,PtrLoadOpLowering 是 swizzle 元信息真正产生运行时效果的地方.指针类型中携带的 SwizzleAttr 在此处被读取,用于变换指针地址或计算 swizzle offset.

两条路径:按地址空间分流

根据指针的地址空间,PtrLoadOpLowering 走两条完全不同的 lowering 路径:

地址空间Lowering 方式Swizzle 处理
BufferDesc (全局显存) ROCDL::RawPtrBufferLoadOp 通过 BufferFatPtr::swizzleByteOffset() 计算 swizzle 后的 byte offset,作为 buffer load 的参数
其他 (shared memory 等) LLVM::LoadOp 通过 applySwizzleOnPtr() 对指针地址做 XOR 变换后再 load

路径 1:BufferDesc -> RawPtrBufferLoad

BufferDesc 地址空间对应 AMD GPU 的 Buffer Resource 机制.指针被表示为一个 BufferFatPtr 结构体,包含:

  • bufferRsrc:4 个 32 位寄存器组成的 buffer 描述符(地址空间 8)
  • valOffset:32 位偏移量(元素为单位)

BufferFatPtr::swizzleByteOffset() 将偏移量转为字节数后,对其应用 XOR swizzle 计算,结果传给 RawPtrBufferLoadOp 的 swizzle offset 参数.硬件在执行 buffer load 时会根据此 offset 做 swizzle 地址变换.

路径 2:其他地址空间 -> applySwizzleOnPtr + LLVM Load

对于 shared memory 等地址空间,swizzle 通过软件实现:applySwizzleOnPtr() 对指针地址做 XOR 变换:

// applySwizzleOnPtr 的核心逻辑:
ptrInt  = ptr_to_int(ptr)
mask    = ((1 << mask_bits) - 1) << (base + shift)
masked  = ptrInt & mask
shifted = masked >> shift
swizzled = ptrInt ^ shifted          // XOR swizzle
result  = int_to_ptr(swizzled)

变换后的指针直接传给 llvm.load.如果 swizzle 是 trivial(mask==0),则 applySwizzleOnPtr 直接返回原指针,不做任何变换.

Swizzle 粒度合法性校验

在 lowering 之前,代码会校验向量 load 的字节大小与 swizzle base 粒度的关系:

int64_t vecBytes = vecTy.getNumElements()
                  * vecTy.getElementType().getIntOrFloatBitWidth() / 8;
int64_t baseBytes = 1 << swizzle.getBase();
if (baseBytes % vecBytes != 0)
  return matchFailure;

SwizzleAttr 有三个参数:mask,base,shift.其中 base 定义了 swizzle 的最小粒度----baseBytes = 1 << base 表示 swizzle 以 baseBytes 字节为最小交换单位,在比这更小的粒度上地址不会被 swizzle 打乱.

校验要求 baseBytes % vecBytes == 0,即 swizzle 粒度必须是向量 load 大小的整数倍.如果向量 load 大小不能整除 swizzle 粒度,load 可能跨越两个被 swizzle 打乱的区域,读取结果不可预测.

合法:baseBytes=16, vecBytes=4
swizzle 粒度: |---16B---|---16B---|
向量 load:    |-4B-|
16 % 4 == 0 OK
load 不会跨越 swizzle 边界
非法:baseBytes=16, vecBytes=24
swizzle 粒度: |---16B---|---16B---|
向量 load:    |------24B------|
16 % 24 != 0 NG
load 跨越 swizzle 边界,
数据被部分打乱!

IR 变换示例 (shared memory 路径)

Before (Fly 方言)
%ptr = fly.make_ptr ...
  : !fly.pointer<f16, shared,
      swizzle=S<2,4,0>>
%val = fly.ptr_load %ptr
  : vector<8xf16>
After PtrLoadOpLowering
; 1. applySwizzleOnPtr:
%ptr_i64 = llvm.ptrtoint %ptr : i64
%mask = arith.constant 0x3F0 : i64
%masked = arith.andi %ptr_i64, %mask
%shifted = arith.shrui %masked, 0
%swizzled = arith.xori %ptr_i64, %shifted
%swizzled_ptr = llvm.inttoptr %swizzled
    : !llvm.ptr<3>

; 2. llvm.load:
%val = llvm.load %swizzled_ptr
    : vector<8xf16>
10 PtrStoreOpLowering
fly.ptr_store -> applySwizzleOnPtr + llvm.store / RawPtrBufferStore

行为:与 PtrLoadOpLowering 完全对称

PtrStoreOpLowering 的逻辑与 PtrLoadOpLowering 完全对称,只是将 load 换成 store:

地址空间Lowering 方式Swizzle 处理
BufferDesc ROCDL::RawPtrBufferStoreOp BufferFatPtr::swizzleByteOffset()
其他 LLVM::StoreOp applySwizzleOnPtr() XOR 变换

swizzle 粒度校验逻辑也完全相同----向量 store 的字节大小必须整除 swizzle base 粒度.

与 PtrLoadOpLowering 的差异

  • Op 消除方式:Load 使用 replaceOp(用 loaded value 替换),Store 使用 eraseOp(store 无返回值,直接删除).
  • 操作数顺序:Store 多一个 value 操作数,作为写入数据传给 RawPtrBufferStoreOp 或 LLVM::StoreOp.

IR 变换示例 (shared memory 路径)

Before (Fly 方言)
%ptr = fly.make_ptr ...
  : !fly.pointer<f16, shared,
      swizzle=S<2,4,0>>
fly.ptr_store %val, %ptr
  : vector<8xf16>
After PtrStoreOpLowering
; 1. applySwizzleOnPtr:
%swizzled_ptr = ... (同 PtrLoad)

; 2. llvm.store:
llvm.store %val, %swizzled_ptr
  : vector<8xf16>, !llvm.ptr<3>

MakeCopyAtomOp 定义

def Fly_MakeCopyAtomOp : Fly_Op<"make_copy_atom", [Pure]> {

After MakeCopyAtomOpLowering
; 无状态 -> llvm.undef:

; 有状态 -> 默认状态:
%atom = ... : !llvm.struct<...>
  ; (由 copyOp.getDefaultState 生成)

16 CopyAtomCallLowering
fly.copy_atom_call -> 实际搬运指令(由 CopyAtomType.emitAtomCall 生成)

CopyAtomCall 定义

def Fly_CopyAtomCall : Fly_Op<"copy_atom_call"> {
  let arguments = (ins Fly_CopyAtom:$copyAtom,
                       Fly_MemRef:$src, Fly_MemRef:$dst,
                       Optional<Fly_MemRef>:$pred);

执行数据搬运操作.CopyAtom 的类型信息决定了具体的搬运方式(如 buffer_load、ds_read、flat_load 等),src/dst 是源和目标的 MemRef,pred 是可选的条件掩码.

Lowering 逻辑

Lowering 委托给 CopyAtomType::emitAtomCall(),该方法根据 CopyAtom 的具体类型生成对应的底层指令:

rocdl.buffer.store %val, %rsrc, %idx ...

MmaAtomCallSSA 定义

def Fly_MmaAtomCallSSA : Fly_Op<"mma_atom_call_ssa"> {

                       AnyType:$a, AnyType:$b, AnyType:$c);
  let results = (outs Variadic<AnyType>:$results);
}

与 MmaAtomCallLowering 的区别在于:SSA 版本的 a/b/c 是 SSA 值(如 vector),d 可以是指针(SSA 值形式)或省略,结果以 SSA 值返回.

Lowering 委托给 MmaAtomType::emitAtomCallSSA().如果有返回值则 replaceOp,否则 eraseOp.

Before (Fly 方言)
%acc = fly.mma_atom_call_ssa %atom, %a, %b, %c
  : (!fly.mma_atom<..., mfma_f32_16x16x16f16>,
     vector<4xf16>, vector<4xf16>, vector<4xf32>)
  -> vector<4xf32>
After MmaAtomCallSSALowering
; 由 emitAtomCallSSA 生成:
%acc = rocdl.mfma.f32.16x16x16f16
  %a, %b, %c : vector<4xf32>

示意 IR:SSA MFMA 的 intrinsic 名与 accumulator 布局由具体 MmaOp 类型决定.

9. GPU 运行时与辅助类 Conversion

20 GpuLaunchFuncOpLowering
gpu.launch_func -> gpu.launch_func(操作数类型已转换)

问题:host 端与 kernel 端类型不匹配

gpu.launch_func 是 host 端调用 GPU kernel 的 op,不属于 Fly 方言.但它的 kernel_operands 可能仍然是 Fly DSL 类型(如 !fly.pointer),而 kernel 内部经过 FlyToROCDL 转换后参数类型已变为 LLVM 类型(如 BufferFatPtr),导致类型不匹配:

Before (类型不匹配)
; kernel 内部(已转换):
gpu.func @my_kernel(
  %arg0: {!llvm.ptr<8>, i32}  <- 期望 BufferFatPtr
)

; host 端(未转换):
gpu.launch_func @my_kernel
  kernel_operands(
    %buf : !fly.pointer<f32, buffer_desc>  <- 传的 Fly 类型
  )

解决:用已转换的操作数重建 op

GpuLaunchFuncOpLowering 用 adaptor(已由 FlyTypeConverter 转换过的操作数)重建 gpu.launch_func,使 host 端操作数类型与 kernel 内部一致:

Before (Fly 方言)
gpu.launch_func @my_kernel
  kernel_operands(
    %buf : !fly.pointer<f32, buffer_desc>
  )
  grid_size(%gx, %gy, %gz)
  block_size(%bx, %by, %bz)
After GpuLaunchFuncOpLowering
gpu.launch_func @my_kernel
  kernel_operands(
    %buf : {!llvm.ptr<8>, i32}  <- 类型已对齐
  )
  grid_size(%gx, %gy, %gz)
  block_size(%bx, %by, %bz)

关键说明

  • op 逻辑不变:grid/block 大小、kernel 引用、动态共享内存大小等全部保留,唯一变化是操作数类型
  • 类型转换由 FlyTypeConverter 统一完成:本 Lowering 只是消费转换后的结果,本身不做类型转换;具体类型映射规则见第 2 节
  • 异步语义处理:支持两种异步模式----asyncObject 和 asyncTokenType + asyncDependencies,二者互斥
  • cluster 大小:如果原始 op 有 cluster 维度信息,也会保留到新 op 中
21 ExtractAlignedPointerAsIndexLowering
fly.extract_aligned_pointer_as_index -> llvm.addrspacecast / 透传

注:源码中该 pattern 带有 // TODO: deprecated in the future 标记,后续可能移除.

ExtractAlignedPointerAsIndexOp 定义

def Fly_ExtractAlignedPointerAsIndexOp
  : Fly_Op<"extract_aligned_pointer_as_index", [Pure]> {
  let arguments = (ins Fly_MemRef:$source);
  let results = (outs AnyType:$result);
}

从 fly.memref 中提取底层裸指针.在 Fly DSL 中,MemRef 是带 layout/swizzle 等元信息的复合类型,但某些场景(如构造 Buffer Resource)需要拿到原始指针值.

Lowering 逻辑

经过 FlyTypeConverter 转换后,fly.memref 已变为 !llvm.ptr<AS> 裸指针.因此提取操作本身很简单----源操作数已经是裸指针了.唯一需要处理的是地址空间不匹配:

  • 源指针的地址空间与结果类型期望的地址空间相同 -> 直接透传
  • 地址空间不同 -> 插入 llvm.addrspacecast 进行地址空间转换
Before (Fly 方言)
%base = fly.extract_aligned_pointer_as_index %memref
  : !fly.memref<f16, shared> -> !llvm.ptr<0>
After ExtractAlignedPointerAsIndexLowering
; memref 已被 FlyTypeConverter 转为 llvm.ptr<3>
; 结果期望 llvm.ptr<0>,地址空间不同
; -> 插入 addrspacecast:
%base = llvm.addrspacecast %memref
  : !llvm.ptr<3> to !llvm.ptr<0>

典型使用场景

在构造 AMD GPU Buffer Resource 时,需要将 shared memory 或 global memory 的指针提取出来作为 base 地址传入 rocdl.make.buffer.rsrc.此时需要从 fly.memref 中提取裸指针,并可能需要转换到通用地址空间(AS=0).
